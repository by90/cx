# cx 通用能力优先复用

## 目的

让人工智能代理先找到项目已经存在的通用能力，再决定是否新增实现。通用能力必须小而稳定，并在 `docs/cx/docs/` 中拥有独立专题文档。

## 强制发现顺序

1. 读取 `docs/cx/docs/00.索引.md` 和当前主题相关的专题文档。
2. 从专题文档取得通用包、公共接口、源码位置、调用方式和适用范围。
3. 搜索 `src/`、当前用例、设计、原任务和真实调用点，核对文档与实现一致。
4. 搜索相邻领域是否已经存在同构结构或可直接复用入口。
5. 记录候选、采用或拒绝理由以及对现有调用点的影响。
6. 只有现有能力确实不能满足当前目标时，才设计新入口。

没有专题文档时仍必须搜索源码，不能把文档缺失当成新增重复实现的理由。发现稳定通用能力缺少专题文档时，先使用 `$cx-doc` 补齐当前说明。

## 调用模型

新增或修改通用能力前，必须明确：

```text
解决的问题：
专题文档：
功能入口：
常规调用方式：
特殊场景入口：
实例或状态生命周期：
状态来源：
真实调用点：
错误策略：
验证方式：
非目标：
```

判断边界：

1. 抽象是否真正隐藏调用复杂度。
2. 调用方是否因此少知道一件事。
3. 是否已有两个真实调用点，或确实隔离了稳定且容易出错的复杂度。
4. 能否直接使用数组、张量、标准切片、构造函数、配置默认参数或库原生错误。
5. 新入口是否会引入当前目标不需要的协议、包装、注册表、调试或未来扩展。

如果直接实现更短、更清楚，就不新增通用层。

## 何时形成通用包

至少满足以下一项：

- 相同逻辑已经有两个或更多真实调用点。
- 数据结构表达稳定项目概念，并且会被多个任务使用。
- 能力跨越多个项目区域，例如训练、数据处理和界面。
- 能力容易出错且需要统一语义，例如索引序列、张量组批、状态归约或检查点路径。

只有一个模糊用例、单行转调、一次性校验或未来猜测时，不抽取。

## 当前状态文档

- 每个通用包或稳定公共接口在 `docs/cx/docs/` 中有一份独立编号专题文档。
- 专题文档说明当前问题、入口、输入输出、约束、最短用法和验证方式。
- 技术路线变化时直接更新原专题文档，不建立新旧版本或迁移历史。
- 包内说明如确有必要，只链接专题文档和列出最短入口，不复制详细内容。

## 实现纪律

- 默认使用满足当前目标的最少代码，不提前框架化、泛化或增加兼容层。
- 所有公共入口和调用方只保留当前最新意图；删除旧接口、别名、适配层、桥接层、兼容参数、兼容配置、兼容路径、旧行为和相关文档痕迹。
- 状态、生命周期和不变量使用明确对象或等价类型表达。
- 默认参数写在函数签名或配置对象中。
- 除非用户在当前请求中明确要求具体校验或异常处理，否则不自行增加校验后抛出异常，不捕获、转换、包装、吞掉、静默跳过或兜底异常；底层异常原样抛出并中止程序。
- 性能路径不做逐条合法性检查、静默过滤或批处理兜底。
- 删除没有真实调用点、没有降低复杂度或只为测试存在的入口。
- 禁止动态属性访问、猴子补丁、动态注入和字符串分发，除非没有静态接口可用并记录理由。

## 必须输出

- 已读取的专题文档。
- 搜索证据和候选实现对比。
- 采用现有入口或新增入口的决定与理由。
- 当前专题文档和源码入口。
- 验证方式。
- `$cx-review` 的统一审查结论。
