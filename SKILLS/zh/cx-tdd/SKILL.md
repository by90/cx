# cx 显式 TDD 实现纪律

## 语言规则

- 使用中文输出对话、说明、计划、总结、审查结论、验证证据和 cx 文档（本工作流维护的用例、任务、变更等文档）时，必须使用完整中文表达，不得写成中英文混杂的半句或词组堆叠。
- 只有代码标识符、命令、路径、接口名、库名、协议、标准名、外部专名，或翻译后会造成歧义的术语可以保留英文；保留英文时，必须在同句或相邻句用中文详细解释其含义、作用和当前上下文。

## 目的

在 `$cx-story` 已经建立主成功场景、当前变更和任务文档，且当前请求或文档明确声明需要 TDD、单元测试、失败测试或 red-green-refactor 后使用本 skill。默认实现路径不使用本 skill，也不主动创建单元测试。

## 必须执行的流程

1. 先确认当前请求、任务文档或变更文档已经明确声明需要 TDD、单元测试、失败测试或 red-green-refactor；没有明确声明时，返回 `$cx-workflow` 的默认单代码文件实现路径。
2. 阅读当前主成功场景文件夹中的 `00.用例.md`、`00.设计.md`、当前 `changes/*.md` 和当前 `tasks/NN.中文任务名.md`。
3. 如果当前变更包含顺序任务清单，只执行当前未完成的第一项；不要提前执行后续项。
4. 确认当前任务的量具是一个类或一组高度内聚的类型；量具不清楚时先更新任务文档并等待确认。
5. 确认任务只绑定一个生产代码文件和一个对应单元测试文件；超出范围必须先拆任务或在任务文档和变更文档中写明理由。
6. 确认提示词或任务文档提供了验证命令，或从仓库中推断最窄的现有命令；两者都做不到时，先询问再实现。
7. 通用功能、可复用功能或可复用类先写公共调用方式测试，再写特殊入口测试、生命周期或状态恢复测试，最后才考虑内部边界。
8. 按项目语言规则放置源文件和测试文件；Python 默认 `src/<subsystem>/xx.py` 对应 `tests/<subsystem>/xx_test.py`。
9. Python 项目的 `src/`、`tests/` 及其所有子目录必须有空白 `__init__.py`；测试和源码都使用从仓库根包开始的绝对导入，例如 `from src.config.config import Config`，不得通过修改 `sys.path` 解决导入问题。
10. 明确声明 Python 单元测试时，VS Code 测试视图必须能发现并运行测试；`unittest` 发现参数使用 `-v -s ./tests -p *_test.py -t .`。
11. 先写最窄的失败测试。
12. 运行测试并记录预期 red failure。
13. 写能让测试通过的最小生产代码。
14. 运行最窄测试直到 green。
15. 只有 green 以后才重构，并且重构期间保持测试 green。
16. 变更影响共享行为时，运行更宽的验证。
17. 代码写完并通过明确要求的测试后，必须使用 `$cx-review` 做代码交付物 review，核对文档一致性、重复味道、完整 OOP、极简实现和业务语义。
18. review 不通过时，当前任务不得标记完成；先修复实现或文档，再重新运行测试和 review。
19. 在当前任务文档和变更文档中记录命令、结果、`$cx-review` 结论、`$cx-evidence` 证据审查结论和残留缺口。
20. review 和证据审查通过后才能更新任务文档、变更文档和验证证据为完成；默认停止在当前任务和当前代码文件边界，用户明确要求连续推进时才继续下一任务或变更项。

## 完整 OOP 规则

- 开发默认完整面向对象；只要存在状态、生命周期、不变量、领域概念或对象协作，就必须用类、dataclass、protocol、struct、enum、trait 或显式方法表达。
- Python 无状态纯计算可以使用函数，但一旦涉及配置、状态、校验策略、领域对象或协作边界，必须建模为类或对象组合。
- Rust 必须用 struct、enum、trait 和明确方法表达领域状态和行为边界。
- 任务量具是类或类型组；任务文档必须写明本任务覆盖哪个类、哪些协作类型，以及为什么必须一起实现。
- 不允许用松散字典、魔法字符串、未命名 tuple 或字符串分发代替明确领域类型。
- 默认禁止 Python `getattr`、`setattr`、`delattr`、monkey patch、动态注入方法或字符串分发。
- 如果确实需要动态反射，必须先证明没有更清晰的静态 API，记录理由，增加针对性测试，并把它隔离在小适配器里。

## 最小实现纪律

铁律：绝对禁止屎山代码。所有源码、测试、脚本、工具、示例代码或流程生成的代码都适用；TDD 不能成为固化臭长入口、无用测试和未来扩展包装的理由。

- 默认选择能满足当前任务的最少代码实现，不提前框架化、泛化或抽象。
- 若实现把本可由少量字段、直接数组切片、标准库语义或一个清晰构造函数完成的能力写成几百行甚至上千行，当前任务必须保持未完成；先删除到最小功能入口，再谈测试通过。
- 文件、类、方法和变量命名要短而清楚；避免把完整句子写进标识符，命名过长时优先提炼职责或复用领域术语。
- 不为单行转调、一次性逻辑或没有真实复用价值的流程创建函数、类、常量或校验器。
- 除非业务规则明确要求，不掩盖任何异常，不静默兜底，不把会导致产品问题的错误转成默认值、空结果、跳过、重试成功假象或 warning；开发期间应让这类错误中止运行。
- 判断标准很简单：如果同样错误发到产品会造成问题，就必须暴露为失败；只有业务明确要求降级、恢复或用户可见提示时，才允许写显式处理路径，并用测试覆盖。
- 不新增“看起来更安全”但需求没有要求的验证，例如文件名白名单、路径合法性检查、额外 AST 扫描、重复配置规则检查等。
- 严禁在大循环、训练循环、热路径或批处理内逐条检查数据合法性来兜底或拖慢性能；数据合法性应在入口、数据准备、测试夹具或单独诊断任务中处理，且不能替代真实失败。
- 默认不自行捕获或包装异常；底层库已经能给出清楚异常时，让原始异常直接抛出。
- 不创建自定义异常类型，除非调用方确实需要区分该异常并已有明确处理路径。
- 默认值优先通过函数、构造参数或配置对象表达，不把简单路径、文件名或一次性默认值提升为模块级常量。
- 只保留当前任务需要的公开 API；不要增加调试入口、内存校验入口、扫描入口或预防未来需求的接口。
- 不为调用方已经能直接完成的动作提供包装，例如数组长度、字段数量、标准切片、复制数组、构造新对象、默认参数传递、库原生错误和普通索引语义。
- 每个 helper 必须同时满足：有明确名字、减少重复或隔离真实复杂度、调用点不止一个或能显著提升可读性。否则内联。
- 重构目标是删除代码、降低分支、缩小公开面，而不是把逻辑搬到更多小函数里。

## 代码质量规则

- 严禁堆砌式混乱代码；同时禁止为单行转调或一次性逻辑制造空壳函数、类或接口。
- 单元测试也必须极简；只固定当前用例要求的行为，不为便利包装、负索引兼容、复制方法、重建方法、补齐方法、兜底校验或未来扩展入口建立测试保护。
- 公共 API 必须小，并通过测试体现文档作用。
- 通用功能、可复用功能或可复用类的测试必须优先证明真实调用方入口正确；禁止先围绕内部 helper 写测试来固化错误设计。
- Python 源码和单元测试都必须包含文件级说明、类职责说明、函数职责说明和逐行意图注释；文件级说明必须说明本文件的作用，以及本文件维护的主要类、函数或测试对象。
- 函数、方法和测试方法的说明必须解释参数含义、返回值或明确无返回值；即使测试方法只有 `self`，也要说明 `unittest` 自动提供该参数。
- 单元测试同样执行逐行意图注释标准；除空行、纯格式行或重复结构外，每一行测试业务逻辑都要有相邻说明注释。
- 修改 Python 源码或测试后必须运行 Black 默认规范检查，例如 `python -m black --check src tests tools`；项目已有更窄路径时可以缩小范围，但不能省略 Black。
- 单元测试必须镜像源码结构，测试文件名必须与源文件一一对应并追加 `_test.py`；不要用一个大测试文件覆盖多个源文件，也不要为同一个源文件创建多个随意命名测试文件。
- `src/`、`tests/` 及其所有子目录必须有空白 `__init__.py`，项目内引用必须使用从仓库根包开始的绝对导入，测试不得修改 `sys.path`。
- VS Code 测试视图必须能发现并运行 `unittest`；推荐发现命令为 `uv run python -m unittest discover -v -s ./tests -p "*_test.py" -t .`。
- 构造函数和普通函数优先使用类型标注与默认参数表达默认行为；配置默认值应直接写成默认参数，例如 `path=Config.default_config_file()` 或 `batch_size=config.train.batch_size`，函数体内用同名字段承载参数，例如 `self.batch_size = batch_size`；不要在 `__init__` 里堆叠大量 `None` 分支、重复命名或参数情况判断，复杂默认构造应拆到 dataclass、配置对象、factory 或小型专用方法。
- Python 脚本禁止使用命令行参数改变行为；不要新增 `argparse`、`click`、`typer`、`sys.argv` 或自定义命令行解析。
- 代码必须极简、短小、直接，避免臃肿文件和重复逻辑。任何可能复用的功能、类或逻辑都必须先调用 `$cx-common-module` 搜索已有实现和设计公共入口。
- `src/<subsystem>/` 下的通用包必须提供包内 `readme.md`，说明公共接口和使用方式；不要把实例配置节、内部字段或实现步骤逐项写成接口文档。
- 最终输出前，按提示词契约审查 diff：目标是否达成、约束是否遵守、验证是否运行、review 是否通过、残留风险是否说明。
- review 不通过时，输出修复结果或剩余发现，不得宣称任务完成。

## 任务映射格式

```text
Story: docs/cx/01.创建用户/00.用例.md
Change: docs/cx/01.创建用户/changes/调整请求校验规则.md
Task: docs/cx/01.创建用户/tasks/01.编写请求校验.md
Gauge: CreateUserRequest 类
Source: src/users/create_user.py
Test: tests/users/create_user_test.py
Expected red: 当前请求对象尚未拒绝缺失用户名
Command: uv run python -m unittest tests.users.create_user_test.CreateUserRequestTest.test_rejects_missing_name
```

## 输出

每个 TDD 循环都应该留下：

- 当前 story、变更文档和任务文档的映射。
- 当前任务量具。
- 已记录的 red failure。
- 最小实现。
- green 测试结果。
- 如发生重构，记录重构说明。
- `$cx-review` 代码交付物 review 结论和 `$cx-evidence` 证据审查结论；FAIL 时任务仍为未完成。
- 更新后的任务文档、变更文档和验证证据。
- 本轮执行模式，以及是否需要在当前任务后等待用户核对。
