# cx PyTorch 全量调参

## 语言规则

- 使用中文输出对话、说明、计划、总结、审查结论、验证证据和 cx 文档（本工作流维护的用例、任务、变更等文档）时，必须使用完整中文表达，不得写成中英文混杂的半句或词组堆叠。
- 只有代码标识符、命令、路径、接口名、库名、协议、标准名、外部专名，或翻译后会造成歧义的术语可以保留英文；保留英文时，必须在同句或相邻句用中文详细解释其含义、作用和当前上下文。

## 资源安全前置门禁

任何全量调参实验的第一步都是评估预计系统内存占用和显存占用，再决定是否启动。评估至少记录当前可用系统内存、当前可用显存、预计峰值、保留给操作系统和 Codex App 的安全余量，以及允许启动或必须调整批次等训练参数的结论；可以使用模型结构、张量形状、批量大小和最小试运行取得估算，但不得先启动完整数据训练再观察是否耗尽资源。全部样本是全量调参的数据边界，资源不足时不得缩小样本比例或改变轻量阶段冻结的参数；允许调整的训练参数仍不能安全运行时，必须等待可用资源。

全量调参属于预期资源占用较多且耗时长的任务，必须在 Codex App 之外的外部终端启动，使训练进程和大量输出不由 Codex App 内置终端承载。训练进度、标准输出、标准错误、系统内存、进程内存、显存占用、每轮指标和错误必须持续写入文件型日志；人工智能代理只通过读取这些日志文件分析实验状态，不依赖终端滚动输出、终端控制序列或其它特殊终端处理。此门禁用于避免系统资源耗尽和系统崩溃，尤其避免 Codex App 因训练进程或终端输出负载而崩溃。

## 目的

把轻量调参确定的 `5` 个候选放到全部样本上验证。全量调参保持候选的数据参数、标签、窗口和模型结构不变，只调整 batch size、learning rate、optimizer 参数和 scheduler 参数；全部候选完成训练、测试和回测后，再选择最佳 release 候选。

## 铁律

1. 全量调参必须以 `$cx-pytorch-quick-hpo` 交付的 `5` 个候选及其完整 recipe、候选确定后的 `30`、`40`、`50`、`60` 天长窗口扩展比较、消融结果和回测结果为输入；不得跳过轻量调参，也不得在全量阶段新增、删除或替换候选身份。
2. 训练脚本必须能单独运行，并在默认配置下完成一次训练；全量调参脚本只能通过 typed config hook 在内存中复制并修改配置，再把该配置对象传给训练入口。
3. 回测脚本必须能单独运行，并在默认配置下完成一次回测；全量调参脚本只能通过内存配置对象调用回测入口，不能直接修改当前回测配置文件。
4. 调参脚本保存结果时，只能把 trial 配置、full-data recipe、测试指标、回测配置和模型产物路径持久化到输出目录；禁止直接改写项目当前配置、默认配置或用户手动维护的配置文件。
5. 全量候选 trial 固定使用全部样本，明确训练 `120` 个 epochs；early stopping 监控 `val_loss`，默认 patience 为 `20` 轮。使用学习率升温和降温循环时，必须显式指定并保存 scheduler 名称、升温轮数、降温轮数、循环次数、最高学习率、最低学习率和变化函数；只有单个完整循环超过 `20` 轮时才可把 patience 调整到足以覆盖一个完整循环，并应优先缩短循环或选择不超过 `30` 轮的 patience，确实必须超过 `30` 时记录不可缩短的理由和实际值。所有 trial、模型初始化、数据打乱和回测中的随机过程统一读取 config 子系统中值为 `3407` 的种子配置。
6. 全量调参冻结轻量阶段确定的数据参数、字段组合、分级边界、特征模式、标签定义、窗口长度、模型结构和模型容量；每个候选只允许搜索 batch size、learning rate、optimizer 参数和 scheduler 参数，不能重新承担轻量调参的选择。
7. 全量正常候选必须跑满 `120` 个 epochs，并在第 `101` 至 `120` 轮内至少出现一次按同一 `val_loss` 改进判定成立的新最佳值。只有全部 `5` 个候选都不满足时，才启用回退排序：先按实际完成 epoch 数从多到少排序，再依次比较最后 `20` 轮的 `val_loss` 改进次数、最佳轮次距结束轮次的距离、best/final `val_loss`、`test_loss` 和回测结果；不得让单次高回测越过训练轮数更少的候选。
8. `5` 个候选都必须使用全部样本完成训练和测试集评估，不得只训练排名靠前的部分候选，也不得在首个候选后等待额外确认。
9. 每个候选完成训练和测试后都必须立即运行回测；回测好坏只影响最终比较，不能作为跳过其余候选的理由。
10. 某个候选训练、测试或回测表现不好时，必须记录该候选的结果和原因，然后继续处理其余候选；全部 `5` 个候选完成后再决定是否回到轻量调参。
11. 最佳模型作为 release 候选保存时，必须遵守现有 `$cx-version` 流程、用户确认门禁、版本记录和发布证据要求。
12. 多模型项目必须先在每个模型内部选出模型内最优 release 候选，再用这些模型内最优方案比较出全局最优；全局最优默认用于推理。
13. 推理必须保留选择不同模型 release 的权利；根目录 `main` 终端菜单和每个模型 `main` 终端菜单都必须提供默认全局最优和可选历史/候选 release 的选择入口。
14. 全量调参必须沿用轻量调参的五分钟跟踪节奏：启动后立即做一次基线采样，运行中每 5 分钟汇报当前候选、当前 trial、当前 epoch、最近 `val_loss`、当前 best `val_loss`、`val_loss` 改进次数、候选已耗时、平均 epoch 耗时、预计剩余 epoch、GPU 利用率、显存占用、显存峰值、系统内存和当前进程内存；如果 5 分钟内没有新 epoch 或新产物，必须说明停留位置、进程号和资源日志路径。
15. 全量调参每个 epoch、每个 trial 和每个候选都必须写入统一资源监控产物。记录至少包含开始/结束时间、epoch 耗时、累计候选耗时、平均 epoch 耗时、GPU 利用率、显存已分配/已保留/峰值/剩余、系统总内存/可用内存、进程内存、样本吞吐、是否 OOM、best/final `val_loss`、`test_loss`、早停 epoch、`val_loss` 改进次数、最佳轮次距结束轮次的轮数和比例，以及资源日志路径。
16. 全量五候选比较和 release 建议必须包含 Top10 业务指标：Top10 命中率、命中候选平均涨幅、未命中候选平均涨幅、Top10 下跌占比和 Top10 下跌平均跌幅。候选表必须同时包含候选身份、允许调整的训练参数、`val_loss`、`test_loss`、训练 epoch、`val_loss` 改进次数、最佳轮次距结束轮次的轮数和比例、trial 耗时、平均 epoch 耗时和上述业务指标。

## 必须执行的流程

1. 读取轻量调参交付的 `5` 个候选、完整 recipe、候选确定后的 `30`、`40`、`50`、`60` 天长窗口扩展比较、逐候选消融结果、逐候选回测结果、字段建议、最终窗口长度、标签定义、模型结构和收敛证据，并锁定不能在全量阶段变化的参数。
2. 用 `$cx-common-module` 检查 full-data recipe、模型保存、测试评估和回测调用是否已有功能入口；没有时先定义调用模型。
3. 用 `$cx-pytorch-tdd` 验证完整数据配置构造、内存 config override、训练入口、测试阶段入口、回测入口和输出持久化。
4. 启动候选前确认资源监控、五分钟采样和五候选汇报字段已经可写入结构化产物；缺失时先补公共监控和汇总入口。
5. 对每个候选只搜索 batch size、learning rate、optimizer 参数和 scheduler 参数；所有比较固定全部样本、明确 `120` 个 epochs、默认早停忍耐 `20` 和配置种子 `3407`。使用升温和降温循环时，trial recipe 必须明确 scheduler 名称、升温轮数、降温轮数、循环次数、最高学习率、最低学习率、变化函数和实际 early stopping patience。
6. 依次用全部样本训练 `5` 个候选并执行测试集评估；每 5 分钟汇报当前 epoch、`val_loss`、改进次数、候选耗时和资源占用，训练结束后记录 `val_loss`、`test_loss`、资源指标和持续性指标。
7. 每个候选训练完成后，用同一份内存配置派生回测配置并立即回测，把回测配置、Top10 命中率、命中均涨幅、未命中均涨幅、下跌占比、下跌平均跌幅和结果保存到输出目录。
8. 全部 `5` 个候选训练、测试和回测完成后，只让跑满 `120` 个 epochs 且第 `101` 至 `120` 轮仍有 `val_loss` 新最佳值的候选正常进入排序；全部候选都不满足时按第 7 条回退。候选集合确定后，再按 `val_loss`、`test_loss`、Top10 命中率、未命中平均涨幅、下跌占比、下跌平均跌幅和业务风险综合排序。
9. 对多模型项目，先为每个模型选出模型内最优候选，再用模型内最优候选比较出全局最优；把全局最优设为默认推理 release。
10. 更新或要求更新根目录 `main` 终端菜单和每个模型 `main` 终端菜单：默认选中全局最优 release，同时允许用户选择其他模型 release 做推理。
11. 选定最佳模型后，把模型产物、recipe、测试结果、回测结果、五分钟跟踪摘要、资源监控、数据版本、随机种子、全局最优依据、推理默认 release 和 release 建议写入当前 `docs/cx` story 的任务文档或变更文档验证证据。

## 验证证据

- 轻量调参 `5` 个候选、完整 recipe、候选确定后的四次长窗口扩展比较、逐候选消融结果和逐候选回测结果的来源。
- 训练脚本和回测脚本可以在默认配置下独立运行。
- 全量调参脚本只在内存中修改配置，输出目录包含 full-data recipe、测试配置、回测配置和候选结果，当前配置文件未被改写。
- 完整数据边界、训练/验证/测试切分、数据版本和随机种子。
- 全部样本、明确 `120` 个 epochs、默认早停忍耐 `20`、实际 patience 和配置种子 `3407` 的执行情况；使用升温和降温循环时的 scheduler 名称、升温轮数、降温轮数、循环次数、最高学习率、最低学习率和变化函数；以及每个候选的早停 epoch、best `val_loss`、final `val_loss`、`test_loss`、`val_loss` 改进次数、改进比例、最佳轮次、最佳轮次距结束轮次的距离和第 `101` 至 `120` 轮是否出现新最佳值。
- 证明数据参数、标签、窗口和模型结构没有在全量阶段变化，以及只搜索 batch size、learning rate、optimizer 参数和 scheduler 参数的配置差异。
- 每 5 分钟跟踪摘要、采样时间、当前候选、当前 epoch、最近 `val_loss`、best `val_loss`、候选已耗时、平均 epoch 秒数、预计剩余 epoch、GPU 利用率、显存占用、显存峰值、系统内存和进程内存。
- 每个候选的 trial 总耗时、平均 epoch 耗时、峰值显存、显存余量、GPU 占用率、系统内存、进程内存、样本吞吐、OOM 或异常原因。
- 每个候选的 Top10 命中率、命中均涨幅、未命中均涨幅、Top10 下跌占比和 Top10 下跌平均跌幅。
- 全部 `5` 个候选的训练、测试和逐一回测结果；任何候选失败时，记录原因和其余候选继续完成的证据。
- 五候选比较表，必须同时展示允许调整的训练参数、正常准入或回退身份、训练可持续性、`val_loss`、`test_loss`、训练 epoch、最后 `20` 轮的 `val_loss` 改进次数、最佳轮次距结束轮次的轮数和比例、trial 耗时、平均 epoch 耗时和 Top10 业务指标；启用回退排序时另存全部候选均未通过正常准入门槛的证据和完成轮数排序。
- 每个模型的模型内最优 release 候选、全局最优选择依据和默认推理 release。
- 根目录 `main` 终端菜单和每个模型 `main` 终端菜单中，可选择默认全局最优或其他 release 的验证证据。
- 最佳模型的 release 候选路径、recipe、指标、回测结果和 `$cx-version` 后续动作。
