# cx PyTorch 全量调参

## 语言规则

- 使用中文输出对话、说明、计划、总结、审查结论、验证证据和 cx 文档（本工作流维护的用例、任务、变更等文档）时，必须使用完整中文表达，不得写成中英文混杂的半句或词组堆叠。
- 只有代码标识符、命令、路径、接口名、库名、协议、标准名、外部专名，或翻译后会造成歧义的术语可以保留英文；保留英文时，必须在同句或相邻句用中文详细解释其含义、作用和当前上下文。

## 资源安全前置门禁

任何全量调参实验的第一步都是评估预计系统内存占用和显存占用，再决定是否启动。评估至少记录当前可用系统内存、当前可用显存、预计峰值、保留给操作系统和 Codex App 的安全余量，以及允许启动或必须缩小实验的结论；可以使用模型结构、张量形状、批量大小和最小试运行取得估算，但不得先启动完整数据训练再观察是否耗尽资源。

全量调参属于预期资源占用较多且耗时长的任务，必须在 Codex App 之外的外部终端启动，使训练进程和大量输出不由 Codex App 内置终端承载。训练进度、标准输出、标准错误、系统内存、进程内存、显存占用、每轮指标和错误必须持续写入文件型日志；人工智能代理只通过读取这些日志文件分析实验状态，不依赖终端滚动输出、终端控制序列或其它特殊终端处理。此门禁用于避免系统资源耗尽和系统崩溃，尤其避免 Codex App 因训练进程或终端输出负载而崩溃。

## 目的

把快速调参得到的候选配置放到完整数据上验证，选出最优到第 3 名方案，并在训练、测试、回测和用户确认后把最佳模型作为 release 候选保存。

## 铁律

1. 全量调参必须以 `$cx-pytorch-quick-hpo` 的 best recipe 和候选列表为输入；如果跳过快速调参，必须在当前 `docs/cx` story 的任务文档或变更文档中记录理由、风险和替代验证方式。
2. 训练脚本必须能单独运行，并在默认配置下完成一次训练；全量调参脚本只能通过 typed config hook 在内存中复制并修改配置，再把该配置对象传给训练入口。
3. 回测脚本必须能单独运行，并在默认配置下完成一次回测；全量调参脚本只能通过内存配置对象调用回测入口，不能直接修改当前回测配置文件。
4. 调参脚本保存结果时，只能把 trial 配置、full-data recipe、测试指标、回测配置和模型产物路径持久化到输出目录；禁止直接改写项目当前配置、默认配置或用户手动维护的配置文件。
5. 全量调参使用完整数据，计划上限是 600 个 epochs，并启用 early stopping 监控 `val_loss`，patience 一般设为 20；触发早停时必须停止，不得强行跑满 600。
6. 全量调参首先以快速调参最佳 `val_loss` 为参考标准，并辅以 full-data `val_loss`、`test_loss` 和回测结果评估候选。
7. 全量调参目标是找出最优到第 3 名方案；但必须先完整训练排名第 1 的方案，训练流程必须包含测试集评估。
8. 排名第 1 的方案完成训练和测试后，必须立即运行回测；如果回测结果好，再询问用户是否需要训练第 2 名和第 3 名候选进行比较。
9. 如果第 1 名方案回测结果不好，必须记录失败原因，并回到候选排序、字段/标签 recipe 或快速调参证据，而不是直接训练其他候选。
10. 最佳模型作为 release 候选保存时，必须遵守现有 `$cx-version` 流程、用户确认门禁、版本记录和发布证据要求。
11. 多模型项目必须先在每个模型内部选出模型内最优 release 候选，再用这些模型内最优方案比较出全局最优；全局最优默认用于推理。
12. 推理必须保留选择不同模型 release 的权利；根目录 `main` 终端菜单和每个模型 `main` 终端菜单都必须提供默认全局最优和可选历史/候选 release 的选择入口。
13. 全量调参必须沿用快速调参的五分钟跟踪节奏：启动后立即做一次基线采样，运行中每 5 分钟汇报当前候选、当前 trial、当前 epoch、最近 `val_loss`、当前 best `val_loss`、`val_loss` 改进次数、候选已耗时、平均 epoch 耗时、预计剩余 epoch、GPU 利用率、显存占用、显存峰值、系统内存和当前进程内存；如果 5 分钟内没有新 epoch 或新产物，必须说明停留位置、进程号和资源日志路径。
14. 全量调参每个 epoch、每个 trial 和每个候选都必须写入统一资源监控产物。记录至少包含开始/结束时间、epoch 耗时、累计候选耗时、平均 epoch 耗时、GPU 利用率、显存已分配/已保留/峰值/剩余、系统总内存/可用内存、进程内存、样本吞吐、是否 OOM、best/final `val_loss`、`test_loss`、早停 epoch、`val_loss` 改进次数和资源日志路径。
15. 全量候选比较和 release 建议必须包含 Top10 业务指标：Top10 命中率、命中候选平均涨幅、未命中候选平均涨幅、Top10 下跌占比和 Top10 下跌平均跌幅。候选表必须同时包含 `val_loss`、`test_loss`、训练 epoch、`val_loss` 改进次数、trial 耗时、平均 epoch 耗时和上述业务指标。

## 必须执行的流程

1. 读取快速调参的 best recipe、top candidates、字段建议、窗口长度、标签定义、训练超参和收敛证据。
2. 用 `$cx-common-module` 检查 full-data recipe、模型保存、测试评估和回测调用是否已有功能入口；没有时先定义调用模型。
3. 用 `$cx-pytorch-tdd` 验证完整数据配置构造、内存 config override、训练入口、测试阶段入口、回测入口和输出持久化。
4. 启动第 1 名候选前确认资源监控、五分钟采样和候选汇报字段已经可写入结构化产物；缺失时先补公共监控和汇总入口。
5. 用完整数据训练第 1 名候选，并在训练流程中执行测试集评估；每 5 分钟向用户汇报当前 epoch、`val_loss`、改进次数、候选耗时和资源占用，训练结束后记录 `val_loss`、`test_loss`、资源指标和业务指标。
6. 训练完成后用同一份内存配置派生回测配置，运行回测并把回测配置、Top10 命中率、命中均涨幅、未命中均涨幅、下跌占比、下跌平均跌幅和结果保存到输出目录。
7. 如果第 1 名候选回测结果达到目标，停止并询问用户是否训练第 2 名和第 3 名候选；用户确认前不要继续消耗算力。
8. 如果用户确认比较，再依次训练和回测第 2 名、第 3 名候选，并按 `val_loss`、`test_loss`、Top10 命中率、未命中平均涨幅、下跌占比、下跌平均跌幅和业务风险综合排序。
9. 对多模型项目，先为每个模型选出模型内最优候选，再用模型内最优候选比较出全局最优；把全局最优设为默认推理 release。
10. 更新或要求更新根目录 `main` 终端菜单和每个模型 `main` 终端菜单：默认选中全局最优 release，同时允许用户选择其他模型 release 做推理。
11. 选定最佳模型后，把模型产物、recipe、测试结果、回测结果、五分钟跟踪摘要、资源监控、数据版本、随机种子、全局最优依据、推理默认 release 和 release 建议写入当前 `docs/cx` story 的任务文档或变更文档验证证据。

## 验证证据

- 快速调参 best recipe 和 top 3 候选来源。
- 训练脚本和回测脚本可以在默认配置下独立运行。
- 全量调参脚本只在内存中修改配置，输出目录包含 full-data recipe、测试配置、回测配置和候选结果，当前配置文件未被改写。
- 完整数据边界、训练/验证/测试切分、数据版本和随机种子。
- 600 epochs / patience=20 的执行情况、早停 epoch、best `val_loss`、final `val_loss`、`test_loss` 和 `val_loss` 改进次数。
- 每 5 分钟跟踪摘要、采样时间、当前候选、当前 epoch、最近 `val_loss`、best `val_loss`、候选已耗时、平均 epoch 秒数、预计剩余 epoch、GPU 利用率、显存占用、显存峰值、系统内存和进程内存。
- 每个候选的 trial 总耗时、平均 epoch 耗时、峰值显存、显存余量、GPU 占用率、系统内存、进程内存、样本吞吐、OOM 或异常原因。
- 每个候选的 Top10 命中率、命中均涨幅、未命中均涨幅、Top10 下跌占比和 Top10 下跌平均跌幅。
- 第 1 名候选的训练、测试和回测结果，以及是否询问用户继续训练第 2/第 3 名候选。
- 第 2 名和第 3 名候选的训练比较结果；如果未训练，记录用户决定或停止理由。
- 候选比较表，必须同时展示 `val_loss`、`test_loss`、训练 epoch、`val_loss` 改进次数、trial 耗时、平均 epoch 耗时和 Top10 业务指标。
- 每个模型的模型内最优 release 候选、全局最优选择依据和默认推理 release。
- 根目录 `main` 终端菜单和每个模型 `main` 终端菜单中，可选择默认全局最优或其他 release 的验证证据。
- 最佳模型的 release 候选路径、recipe、指标、回测结果和 `$cx-version` 后续动作。
