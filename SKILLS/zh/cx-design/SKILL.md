# cx 面向对象设计

## 语言规则

- 使用中文输出对话、说明、计划、总结、审查结论、验证证据和 cx 文档时，必须使用完整中文表达，不得写成中英文混杂的半句或词组堆叠。
- 只有代码标识符、命令、路径、接口名、库名、协议、标准名、外部专名，或翻译后会造成歧义的术语可以保留原文；保留英文时，必须在同句或相邻句用中文解释其含义、作用和当前上下文。

## 目的

使用本 skill 处理实现前的面向对象设计和实现后的设计审查。它必须先确定业务概念、职责边界、对象协作、生命周期和变化原因，再允许进入编码或重构。

本 skill 的目标不是套用设计模式，而是防止把数据库访问、领域规则、字段枚举、集合管理、基础设施、测试便利入口和未来扩展入口混进同一个类。

## 来源基础

- Robert C. Martin 对单一职责原则的解释强调：模块应只有一个变化原因；他引用 Parnas 的模块分解观点，强调模块应隐藏容易变化的设计决策。
- Rebecca Wirfs-Brock 的责任驱动设计强调：先识别关键对象的角色、职责和协作，再决定类的内部细节。
- Martin Fowler 对贫血领域模型的批评强调：只有数据没有领域行为的对象会付出领域模型成本，却失去面向对象组织复杂业务逻辑的收益。

## 触发条件

遇到以下任一情况，必须使用本 skill：

1. 用户要求设计、架构、面向对象、职责拆分、领域对象、类命名、文件拆分或数据访问边界。
2. 一个类或文件同时承担数据库连接、查询、领域规则、字段定义、集合生命周期、基础设施算法、测试便利入口或兼容旧路径。
3. review 发现类太长、方法太多、变量传递过多、校验过多、接口绕远、对象只是包装数据，或代码无法用一句话说明职责。
4. 要新增或调整通用基础设施、领域集合、字段枚举、数据库网关、仓储对象、领域服务或数据容器。
5. 需要决定某个功能应继承、组合、独立成类、放入通用包，还是留在领域包。

## 设计步骤

1. 从用户目标和现有 cx 文档中提取业务名词，不先从技术流程、测试便利或旧代码文件出发。
2. 把候选对象分成领域实体、领域集合、值对象、字段枚举、数据库网关、领域服务、基础数据结构和外部适配器。
3. 为每个候选对象写一句职责说明：它拥有什么数据，维护什么不变量，提供什么行为，因为什么业务变化而改变。
4. 写清楚每个对象不负责什么；这个部分和职责说明同等重要。
5. 写出协作关系：谁创建谁，谁持有谁，谁调用谁，谁只传入依赖，谁必须关闭或释放资源。
6. 检查变化原因：如果一个类会因为两类独立需求变化而变化，就拆分。
7. 检查信息归属：行为应放在最了解所需数据和规则的对象中，不应塞进无关服务或工具函数。
8. 检查数据访问边界：数据库连接、执行命令和资源关闭归数据库网关；领域对象不得直接散落使用底层连接。
9. 检查领域行为：领域规则、业务计算和不变量归领域对象或领域服务；数据库网关不得携带业务规则。
10. 检查基础设施：可复用容器、索引结构和数组设施归通用包；领域读取类可以继承或组合它，但不得把通用设施复制进领域文件。
11. 形成设计结论后，再决定文件、类、构造函数、方法和测试范围。

## 候选对象分类

- 领域实体：表示单个稳定业务对象，负责自身业务属性、不变量和与自身数据强相关的行为。
- 领域集合：表示一组领域实体和市场、账户、用户、订单等整体参数，负责排序、索引、查找、生命周期和跨实体但仍属于该集合的规则。
- 值对象：表示不可变或按值比较的业务值，例如日期、区间、比例、状态或组合键。
- 字段枚举：只负责稳定字段序号、字段含义和数组列访问约束，不负责读取、计算或数据库连接。
- 数据库网关：只负责连接、执行查询、事务、资源释放和数据库错误自然暴露，不负责领域判断。
- 领域服务：只在一个行为天然跨越多个领域对象且没有单个信息专家时使用；它应协调对象完成业务行为，不应变成过程式大函数仓库。
- 基础数据结构：只负责通用存储、索引、切片和性能语义，不负责具体业务字段和业务规则。
- 外部适配器：只负责把外部系统、文件、接口或设备转换成本项目对象能理解的输入。

## 强制门禁

1. 每个类必须能用一句中文说明职责；说不清就不能写代码。
2. 每个类必须能列出不负责事项；列不出说明边界还没有形成。
3. 类名必须来自稳定业务概念或通用基础设施概念，不得用临时流程名掩盖多重职责。
4. 字段枚举、领域集合、领域实体、数据库网关和读取器默认分离；只有职责和变化原因完全一致时才允许合并。
5. 数据库路径、连接、执行命令和资源关闭只能集中在数据库网关或仓储边界。
6. 领域对象不得通过字符串字段名访问结构化字段；需要稳定列序号时使用明确枚举。
7. 性能热路径不得为了“看起来完整”增加大量预检查、兜底转换、兼容入口或静默过滤。
8. 默认参数必须写在函数签名中；函数体内不得堆叠大量 `None` 分支或重复命名变量表达默认行为。
9. 旧接口、别名、适配层和兼容分支在开发期必须删除，除非用户当前明确要求保留。
10. 若一个实现可以用少量字段、直接对象协作和清晰构造函数表达，不得写成几百行模板式代码。

## 反例信号

出现以下信号时，必须暂停编码并回到职责拆分：

1. 一个类同时叫读取器，却又保存市场参数、字段枚举、数据库连接、索引容器和业务规则。
2. 一个字段枚举被藏在读取类内部，导致调用方必须依赖读取类才能访问字段序号。
3. 领域集合和单个领域对象放在同一文件，且二者变化原因不同。
4. 数据库连接散落在多个领域类中，或每个类都自己拼接连接逻辑。
5. 服务类包含所有业务计算，领域对象只剩属性包。
6. 为未来可能用途添加克隆、重建、兼容旧名、负索引、调试入口、批量兜底或测试专用公开方法。
7. 为了通过测试增加非业务入口，而不是让测试使用真实功能入口。
8. 代码注释在解释绕远实现，而不是解释业务含义。

## 审查清单

设计完成或代码完成后，按以下问题逐项审查：

1. 这个类的唯一变化原因是什么？
2. 这个类是否持有了不该由它知道的数据？
3. 这个行为是否放在最了解数据和规则的对象里？
4. 这个对象是否只是数据包；如果是，是否明确它只是边界数据或基础设施数据？
5. 数据库访问是否集中在数据库网关或仓储边界？
6. 字段序号、业务含义、读取逻辑和集合生命周期是否各自有清晰归属？
7. 是否存在为了“完整”而添加的多余方法、多余参数、多余变量、多余校验或未来扩展入口？
8. 实现是否能用最短真实调用路径完成主成功场景？
9. 测试或验证是否覆盖真实功能入口，而不是只验证内部绕路？
10. 设计文档、任务文档、代码文件和使用说明是否说的是同一套对象边界？

## 与其它 skill 的协作

1. 有主成功场景、任务或变更时，先用 `$cx-doc` 读取相关专题文档，再用 `$cx-story` 定位当前用例、设计和原任务，然后使用本 skill 做职责设计。
2. 涉及通用能力、可复用基础设施或共享接口时，在本 skill 明确对象边界后，再用 `$cx-common-module` 检查已登记通用包、已有实现和设计复用入口。
3. 用户明确要求单元测试或 TDD 时，在本 skill 形成设计结论后，再使用 `$cx-tdd` 或对应专项测试 skill。
4. 任一代码、设计、文档或流程交付物完成后，必须使用 `$cx-review` 连续完成交付物质量审查和完成证据门禁。

## 输出要求

使用本 skill 后，最终结论必须包含：

1. 发现的主要设计问题。
2. 拆分后的对象和职责。
3. 明确不做的事项。
4. 需要修改的文件和顺序。
5. 验证方式和 review 结果。
