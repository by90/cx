# cx 流程编排

## 目的

作为 cx 技能入口，先理解项目已经固化的知识和可复用能力，再决定当前工作。流程不保存开发过程的中间文档；正式文档只反映最新状态，变更历史由 Git 保存。

## 压倒一切的原则

1. 不保留开发过程中的中间产物、完成变更档案、旧方案、草稿、备份或并行计划。
2. 项目说明、用例、设计、任务、专题文档和研究笔记只写当前有效状态。
3. 新旧主要差异只写入当前未完成的 `changes/` 文件。
4. 变更文件先提交到 Git；工作完成并审查通过后删除，再提交删除动作。
5. 已有 story 的任务文件数量、编号、文件名和身份保持不变；实现改向和代码错误直接修改原任务。
6. Git 提交、标签和发布记录保存历史，cx 文档不保存历史。

## 启动顺序

1. 对多步任务先在对话中建立可见待办事项列表。
2. 读取项目根目录 `AGENTS.md`；缺失或没有反映项目目标、语言与通用包时，先使用 `$cx-doc` 的 `assets/AGENTS.md` 建立或修正它。
3. 根据当前领域读取项目 `AGENTS.md` 登记的全部通用包教程，再检查 `docs/cx/docs/00.索引.md`。
4. 搜索教程登记的通用包、公共接口、源码位置和真实调用点。
5. 读取 `docs/cx/00.项目说明.md`、目标 `00.用例.md`、`00.设计.md` 和原任务文档。
6. 检查目标主成功场景的 `changes/`。
7. 有未完成变更时按变更文件工作；没有时按原任务当前要求工作。
8. 如果已有 story 发生需求变化、设计改向、实现改向或代码错误，先创建或更新变更文件并提交到 Git。
9. 根据任务类型选择最小技能组合，再执行修改和验证。

## 通用能力优先门禁

在新增任何类、函数、组件、数据结构、协议实现或工具前，必须依次完成：

1. 从项目 `AGENTS.md` 定位当前领域，并读取其中登记的全部通用包教程。
2. 搜索项目内已经登记的通用包和稳定公共接口。
3. 搜索源码和真实调用点，确认文档没有失效。
4. 记录采用现有入口或拒绝候选的具体理由。
5. 只有没有现有能力满足需求时，才设计新入口。
6. 新增通用包时同步创建调用者教程、更新索引和项目 `AGENTS.md` 导航。

对通用包视而不见并新增相似实现，属于重复味道，`$cx-review` 必须判定失败。

## 当前状态与变更

- 新 story 可以建立新的固定任务集合。
- story 建立后不得新增、删除或重命名任务文件。
- 原任务改用另一种方式时，先用 `$cx-changelog` 登记变化，再改写原任务正文。
- 代码错误使用同一流程，不建立修复任务或错误历史文档。
- 正式文档不能出现“之前”“改为”“原方案”“新方案”“迁移过程”等历史叙述；这些内容只允许存在于未完成变更文件。
- `$cx-review` 通过前保留变更文件；通过后删除并提交。

## 专题文档与研究

- 通用包必须使用 `$cx-doc` 维护调用者教程，并在项目 `AGENTS.md` 登记领域、教程链接、公共入口和先读条件。
- 协议、数据过程、特征体系、指标口径和技术转向使用 `$cx-doc` 维护 `docs/cx/docs/`。
- 研究任务使用 `$cx-research`，并把针对具体问题的综合结论保存到 `docs/cx/notes/`。
- 研究笔记先回答问题，再用通俗中文解释依据、限制和工作影响。
- 搜索草稿、材料摘录和候选过程不进入仓库。

## 技能选择

- 用例、固定任务集合和当前 story：`$cx-story`。
- 已有 story 的变化、实现改向或代码错误：`$cx-changelog`，随后回到 `$cx-story` 的原任务。
- 通用包教程、项目 `AGENTS.md` 导航、专题文档、协议、数据过程、特征体系或技术转向：`$cx-doc`。
- 通用功能、可复用类和稳定接口：先 `$cx-doc`，再 `$cx-common-module`。
- 架构、职责、领域对象和数据访问边界：`$cx-design`。
- 研究、模型选择和论文综合：`$cx-research`，并使用 `$cx-doc` 保存研究笔记。
- 用户或当前文档明确声明单元测试或测试先行：先使用 `$cx-tdd` 执行唯一主流程；Python、PyTorch 或 Lightning 测试再叠加 `$cx-pytorch-tdd`，Rust 测试再叠加 `$cx-rust-tdd`。
- Rust 普通实现遵循当前任务的默认实现流程；没有明确测试要求时不使用 `$cx-rust-tdd`。
- 发布版本：`$cx-version`。
- 任一交付物完成后的质量审查和完成证据门禁：`$cx-review`。

## 执行纪律

1. 默认完成当前任务文档并只编辑它绑定的一个生产代码文件。
2. 用户明确要求连续推进多个任务时，按原任务顺序继续。
3. 默认不创建、不修改、不运行单元测试；只有用户、当前任务或当前变更明确声明时才执行。
4. 涉及状态、生命周期、不变量和领域协作时使用完整对象模型。
5. 实现保持极简，删除多余校验、异常兜底、重复变量、无调用入口和未来扩展。
6. 除非用户在当前请求中明确要求具体校验或异常处理，否则禁止自行增加校验后抛出异常，也禁止捕获、转换、包装、吞掉或兜底异常；底层异常保持原始类型、信息和调用栈并自然中止程序。
7. 开发代码只实现当前最新意图；旧接口、别名、适配层、桥接层、兼容分支、旧参数、旧配置、旧路径、旧行为和相关痕迹全部删除，调用方同步使用当前入口。
8. 修改 cx 源仓库时先完成中文源和中文包，再同步英文源和英文包。
9. 修改完成后提交并推送 `main`，再从远端运行安装脚本更新全局技能和 `AGENTS.md`。

## 收尾顺序

1. 更新正式文档和实现为当前状态。
2. 执行最窄有效验证，再执行必要的包级验证。
3. 使用 `$cx-review` 完成交付物质量审查。
4. 使用同一次 `$cx-review` 执行完成证据门禁。
5. 审查失败时保留变更文件并继续修复。
6. 审查通过时删除变更文件，提交删除动作。
7. 确认待办事项全部完成、取消或明确阻塞，并报告验证结果和剩余风险。
