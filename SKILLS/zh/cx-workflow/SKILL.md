# cx 流程处理

## 目的

使用本 skill 作为 cx 工作流入口。它负责判断用户请求属于哪类工作、选择需要组合的 cx skills、安排执行顺序，并在需求不清或风险过高时先澄清。

## 流程入口

1. 先判断任务类型：需求讨论、功能实现、缺陷修复、重构、专项技术实现、文档更新、证据审查或安装使用问题。
2. 任何功能组工作都必须先有独立功能分支，再修改文件。功能分支完成后合并到 `dev`，不能直接合并到 `main`。
3. 识别项目是否已经存在 `docs/INDEX.md` 或 `docs/README.md`、一个或多个研发文档集，以及 `AGENTS.md`。
4. 判断是否会改变代码行为、公共 API、数据结构、用户工作流、发布方式或研究结论。
5. 根据影响范围选择最小必要的 cx skills，不要一次性套用所有 skills。
6. 为请求选择目标文档集；多组功能使用 `docs/<feature-group>/`，单组功能可以继续使用 `docs/` 根文档集。
7. 如果任务涉及通用功能、可复用功能、可复用类、共享工具、配置、日志、路径、缓存、运行环境探测或数据访问入口，先进入“通用/可复用功能门禁”，再允许 BDD/TDD 或实现。
8. 给出当前步骤或直接执行；只有需求缺口会导致错误实现时才先提问。

## 强约束

1. 对需要 BDD、研发主文档、实现计划或变更记录的研发任务，完成文档更新后必须停止，向用户汇报文档结果，并等待用户明确确认；没有确认前不能写测试、不能改实现、不能进入 TDD。
2. 当前项目安装的是中文 cx 包时，所有 cx 生成或维护的文档必须使用简体中文，包括 `BDD.md`、`ENGINEERING_SPEC.md`、`CHANGELOG.md`、`docs/INDEX.md`、计划、测试矩阵和验证证据；代码标识符、命令、API 名称和外部英文专名可以保留原文。
3. 用户要求提交、交付、发 PR 或发布时，把当前工作区视为一次整体变更：暂存已跟踪和未跟踪文件，创建一个提交。不要分析哪些文件是自己修改的、哪些是用户修改的，也不要按来源拆分提交；只有明显的密钥、构建产物或无关大文件才需要先停下来确认。
4. 工作完成后，相关功能分支应合并到 `dev` 并删除本地功能分支。只有用户明确要求时，才允许把功能分支推送到远端；默认只推送 `dev`，以及用户明确要求发布交接时推送 `main`。
5. 使用“已验证依据”或“验证证据”描述已经被文档、测试、命令输出或来源支撑的信息；不要使用“工程事实”这个不直观的词。
6. 任何实现、修复、重构、通用功能、可复用功能、可复用类或可复用能力抽取任务都必须遵守对应实现 skill 中的 `## 最小实现纪律`：绝对禁止屎山代码，默认采用能满足当前需求的最少代码。
7. 通用功能、可复用功能或可复用类实现前，必须先确定调用模型；调用模型未写入 BDD/ENGINEERING_SPEC 前，不允许开始写内部加载、校验、转换、缓存或持久化代码。

## 通用/可复用功能门禁

遇到通用功能、可复用功能、可复用类或共享工具时，先整理并记录以下契约：

```text
公共入口：
常规调用方式：
特殊场景入口：
实例或状态生命周期：
状态来源：
测试如何覆盖所有源码调用：
非目标：
```

- 公共入口说明调用方应该 import、构造或调用什么。
- 常规调用方式说明业务代码最常见、最短的用法。
- 特殊场景入口说明只有测试、不同数据源、不同配置目录或临时上下文才允许使用的入口。
- 实例或状态生命周期说明何时创建、缓存、重置、恢复和释放。
- 状态来源说明配置、文件、环境、数据库、设备探测或用户输入来自哪里。
- 测试覆盖方式说明测试如何让源码中的常规调用也走测试路径，而不是只测特殊入口。
- 非目标明确哪些验证、兼容层、动态扫描、注册表、未来扩展或异常包装本次不做。

## 提示词输入契约

当用户请求含混时，先把它整理成这个契约再规划：

```text
目标：
上下文：
分支：
约束：
必须遵循的流程：
验证方式：
交付物：
```

- 目标必须描述可观察行为，或明确的研究/发布结果。
- 上下文应标明目标文档、文件、分支、环境和相关历史决策。
- 分支应标明功能组分支名、合并目标 `dev`，以及是否请求发布交接到 `main`。
- 约束应包含公开 API、语言规则、兼容性、性能、安全或风格限制。
- 必须遵循的流程要点名最小必要 cx skills。
- 验证方式列出期望命令、检查、截图或证据。
- 交付物说明要修改什么，以及最终摘要/证据要求。

只有缺失字段会造成错误行为、不安全改动或无法验证的工作时，才问澄清问题。否则直接执行，并简短说明假设。

## Skill 选择

- 行为发现、需求、验收标准、业务规则或场景编写：使用 `$cx-bdd`。
- BDD 清楚后的测试先行实现：使用 `$cx-tdd`。
- 行为变更、缺陷修复、架构调整、实现规划：先用 `$cx-bdd`，再用 `$cx-tdd`。
- 只更新变更记录或检查 change ID：使用 `$cx-changelog`。
- 发布版本判断、SemVer 升级、tag 或 release notes：使用 `$cx-version`。
- 模型选择、模型原理、近期 AI 论文、学术/博客综合研究或研究报告：使用 `$cx-research`。
- Python、PyTorch、Lightning、tensor 或 ML 测试：叠加 `$cx-pytorch-tdd`。
- Rust 实现或 Rust 测试：叠加 `$cx-rust-tdd`。
- 抽取或设计通用功能、可复用功能、可复用类、共享工具、稳定 API、复用组件、迁移重复逻辑：叠加 `$cx-common-module`。
- 交付前检查、测试证据、文档一致性：使用 `$cx-evidence`。

## 执行顺序

1. 对普通研发任务，先创建或切换到独立功能组分支，再修改项目文件。
2. 然后走 `$cx-bdd`，选择或创建目标编号功能文件夹，并更新 `BDD.md`、`ENGINEERING_SPEC.md` 和 `CHANGELOG.md`。
3. 文档更新完成后停止，把文档变更和下一步实现计划汇报给用户，等待用户明确确认。
4. 用户确认后，才使用 `$cx-tdd` 和必要专项 skill 写失败测试、记录 red failure、实现最小改动并验证。
5. 功能组完成后，把功能分支合并到 `dev`；不要把功能分支直接合并到 `main`。
6. 对已有实现的收尾任务，先检查文档和测试证据，再使用 `$cx-evidence`。
7. 对仅安装、更新、语言切换或 shskills 使用问题，直接回答命令，不启动 BDD/TDD。
8. 对只读分析或代码审查，先读相关文件并列出风险，不做代码改动，除非用户要求修复。
9. 对跨多个功能组的大改动，先拆成功能组文档集，并在 `docs/INDEX.md` 记录顺序、依赖和状态。
10. 对可能复用的功能、类、组件、数据结构、测试夹具或 UI 状态模型，先使用 `$cx-common-module` 搜索已有实现和 registry，再决定是否新增抽象。
11. 对 progress UI 或 ragged tensor 工具这类组件域，不要新增 cx skill；它们应作为项目组件维护自己的 README 和测试。
12. 对发布工作，使用 `$cx-version`：用户确认版本完成后，才把 `dev` 合并到 `main`；只有 `main` 可用于版本提交、带注释发布 tag 和发布 tag push。功能分支和 `dev` 仍然可以为了协作、备份或 CI 正常 push。

## 停止条件

- 缺少关键业务规则、目标环境或验收标准，并且继续实现会造成明显返工。
- 用户要求的实现与现有 `AGENTS.md`、技术约束或公开 API 冲突。
- 需要联网核对高风险或易变信息，但当前环境无法验证。

遇到停止条件时，先说明阻塞点和需要用户确认的最少问题。
