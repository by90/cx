# cx 证据审查

## 目的

审查工作是否真的由 `docs/cx` 下的 story、任务和变更文档驱动，并且是否通过严格 TDD、严格 OOP 和验证证据支撑。这是交付门槛，不是只看风格的 review。

## 审查清单

1. 所有 cx 场景、任务和流程文档是否都在 `docs/cx/` 下？
2. `docs/cx/` 之外是否没有被当作 cx 流程文档维护的内容？
3. 是否存在 `docs/cx/00.项目说明.md`？
4. 每个主成功场景是否有 `00.用例.md`、`00. 设计.md`、`tasks/` 和 `changes/`？
5. 当前工作是否先检查并处理了 `changes/` 中未完成变更？
6. 如果一个变更涉及多个任务或用例文档修改，是否存在顺序任务清单？
7. 是否记录了本轮执行模式；如果选择逐任务确认，是否在每个任务完成后等待用户核对；如果默认直接做完，是否连续完成文档、测试、实现和验证？
8. 每个任务是否有 `tasks/<编号.任务名>/00.任务.md`？
9. 每个任务是否声明了类或类型组作为任务量具？
10. 每个任务是否只绑定一个代码文件和最多一个单元测试文件；若超出范围，是否写明理由？
11. 条件、替代和异常是否用 `1.1`、`2.1` 这类子编号写在 `00.用例.md` 的对应主成功步骤下，而不是创建分支文件夹或堆叠独立分支章节？
12. 是否没有创建草稿、备份、旧版、并行计划或重复说明文档？
13. 实现前是否展示了预期 red failure？
14. 测试命令和结果是否记录到任务文档或变更文档？
15. 长时间运行、构建、测试、安装或 UI 实机检查是否启动了项目提供的临时防睡眠机制，并在收尾前关闭？
16. UI 相关修改是否按项目方式打包、安装或启动真实应用，并在 macOS GUI 项目中使用 Computer Use 或项目指定方式完成实机观察？
17. 可复用功能、类或组件是否先搜索已有实现、相关 skills 和当前 `00. 设计.md` 中的复用记录，再抽取或说明不抽取理由？
18. Python 测试是否默认使用 `unittest`，除非项目有例外？
19. Python 源码是否位于 `src/<subsystem>/`，测试是否在 `tests/` 下镜像 `src` 结构，并按 `xx.py` -> `xx_test.py` 一一对应？
20. 新增或修改的源码文件和单元测试是否都有文件级作用说明，并说明本文件维护的主要类、函数或测试对象？
21. 新增或修改的类、函数、方法和测试方法是否都有职责说明、参数含义说明、返回值说明或明确无返回值说明？
22. 新增或修改的源码和单元测试是否都遵守逐行意图注释标准，且每一行业务逻辑都有相邻说明注释？
23. Python 修改后是否运行了 Black 默认规范检查，例如 `python -m black --check src tests tools` 或项目等价命令？
24. Python 默认行为是否通过默认参数、配置对象、dataclass、factory 或小方法表达，而不是在 `__init__` 中堆叠大量分支？
25. Python 脚本是否没有新增命令行参数解析，且所有可调整行为都通过 config 子系统配置项表达，并且每个配置项都有默认值？
26. 是否遵守对应实现 skill 的最小实现纪律：绝对禁止屎山代码，未提前框架化、泛化、抽象或增加需求外验证？
27. 是否没有捕获、吞掉、静默跳过、默认值兜底、空结果兜底、假装重试成功或包装任何会造成产品问题的异常；如存在处理，是否有明确业务要求、用户可见路径和测试？
28. 是否没有在大循环、训练循环、热路径或批处理内逐条检查数据合法性来兜底、过滤或拖慢性能？
29. 代码是否保持极简、短小、直接，并且可复用功能、类或逻辑已经先使用 `$cx-common-module` 搜索和登记？
30. 是否采用严格 OOP：Python 用类、dataclass、protocol 或显式对象协作表达状态和不变量；Rust 用 struct、enum、trait 和方法表达领域对象？
31. 是否避免了非 OOP 的动态访问方式，例如 `getattr`、`setattr`、`delattr`、monkey patch、动态注入或字符串分发；如不可避免，是否记录理由、隔离实现并测试？
32. 通用功能、可复用功能或可复用类是否先明确公共入口、常规调用方式、特殊场景入口、实例或状态生命周期、状态来源、测试如何覆盖所有源码调用和非目标？
33. 测试是否覆盖特殊入口、生命周期、状态恢复或测试隔离方式？
34. 是否存在只有一个调用点且没有隔离真实复杂度的 helper、包装函数、校验器或常量？
35. 是否存在需求未要求的验证、异常包装、动态构造、扫描器、注册表或未来扩展入口？
36. Rust 代码变更是否运行了必要 cargo 命令？
37. 工作是否满足提示词契约：目标、上下文、约束、必须遵循的流程、验证方式和交付物？
38. 最终摘要是否报告已运行命令、结果、跳过检查及原因、残留风险？

## 输出格式

```text
Findings:
1. [severity] file:line - issue
   Evidence:
   Fix:

Verified:
- command -> result

Missing evidence:
- ...
```
