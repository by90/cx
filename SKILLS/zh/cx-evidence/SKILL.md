# cx 证据审查

## 语言规则

- 使用中文输出对话、说明、计划、总结、审查结论、验证证据和 cx 文档（本工作流维护的用例、任务、变更等文档）时，必须使用完整中文表达，不得写成中英文混杂的半句或词组堆叠。
- 只有代码标识符、命令、路径、接口名、库名、协议、标准名、外部专名，或翻译后会造成歧义的术语可以保留英文；保留英文时，必须在同句或相邻句用中文详细解释其含义、作用和当前上下文。

## 目的

审查工作是否真的由 `docs/cx` 下的 story、任务和变更文档驱动，并且是否已经对代码、文档、教程、研究、设计、流程变更或发布说明等实际交付物执行 `$cx-review`。这是任务、变更和交付前的证据门槛，不是可选总结。任何 P0/P1/P2 发现、缺少必需 review 或任一必需 review FAIL，都表示证据审查不通过，当前任务、变更或交付物不得标记完成。

## 证据重点

1. Review 完整性：本轮触及的每类交付物是否都有 `$cx-review` PASS 记录；代码、文档、教程、研究、设计和流程变更不能互相替代 review。
2. 文档一致性：任务、变更和 review 结论是否与 `00.用例.md`、`00.设计.md`、当前任务文档、当前变更文档和用户请求一致。
3. 业务语义：交付物是否表达正确业务含义；不能只看起来完整、格式正确或命令能跑。
4. 验证证据：命令、截图、来源、人工检查或其它证据是否真实覆盖当前交付物；单元测试只在明确声明时作为证据来源。
5. 完成状态：没有缺证据、未 review、review FAIL、用户未确认或剩余阻塞的内容被写成已完成。
6. 语言证据：对话摘要、cx 文档（本工作流维护的用例、任务、变更等文档）、审查结论和验证证据是否使用完整中文表达；保留英文时是否有相邻中文解释。

## 审查清单

1. 所有 cx 场景、任务和流程文档是否都在 `docs/cx/` 下？
2. `docs/cx/` 之外是否没有被当作 cx 流程文档维护的内容？
3. 是否存在 `docs/cx/00.项目说明.md`？
4. 每个主成功场景是否有 `00.用例.md`、`00.设计.md`、`tasks/` 和 `changes/`？
5. 当前工作是否先检查并处理了 `changes/` 中未完成变更？
6. 如果一个变更涉及多个任务或用例文档修改，是否存在顺序任务清单？
7. 是否记录了本轮执行模式；默认模式下是否只完成当前任务文档并编辑一个生产代码文件；如连续推进多个任务，是否有用户明确要求？
8. 每个任务是否有 `tasks/NN.中文任务名.md`，且没有 `00.任务.md` 这类泛名任务文件？
9. 每个任务是否声明了类或类型组作为任务量具？
10. 每个任务是否只绑定一个生产代码文件；若超出范围，是否拆成多个任务或写明理由？
11. 条件、替代和异常是否用 `1.1`、`2.1` 这类子编号写在 `00.用例.md` 的对应主成功步骤下，而不是创建分支文件夹或堆叠独立分支章节？
12. 是否没有创建草稿、备份、旧版、并行计划或重复说明文档？
13. 文档是否只写具体事实、具体动作和具体决策，没有空泛铺垫、重复目标、遗漏“要做什么”或自造未定义术语？
14. 项目说明、用例、设计和任务是否各自承担自己的内容，没有把同一套目标重复写到多处？
15. 变更文档是否只记录已经实施后的后续变化，文件名是否没有时间戳？
16. 单元测试或 TDD 是否由用户请求、任务文档或变更文档明确声明；未声明时是否没有创建或修改单元测试？
17. 明确要求测试时，是否展示了预期 red failure 并记录测试命令和结果；未要求测试时，是否记录了其它验证方式？
18. 本轮触及的代码、文档、教程、研究、设计、流程变更或发布说明是否分别有 `$cx-review` PASS 结论？
19. 如果某类交付物没有 review，是否仍保持未完成状态而不是写成完成？
20. 长时间运行、构建、测试、安装或 UI 实机检查是否启动了项目提供的临时防睡眠机制，并在收尾前关闭？
21. UI 相关修改是否按项目方式打包、安装或启动真实应用，并在 macOS GUI 项目中使用 Computer Use 或项目指定方式完成实机观察？
22. 可复用功能、类或组件是否先搜索已有实现、相关 skills 和当前 `00.设计.md` 中的复用记录，再抽取或说明不抽取理由？
23. 明确要求 Python 单元测试时，是否使用 `unittest`，除非项目有例外？
24. Python 源码是否位于 `src/<subsystem>/`；明确要求测试时，测试是否在 `tests/` 下镜像 `src` 结构，并按 `xx.py` -> `xx_test.py` 一一对应？
25. `src/`、`tests/` 及其所有子目录是否都有空白 `__init__.py`？
26. 项目内引用是否使用从仓库根包开始的绝对导入，例如 `from src.config.config import Config`，并且测试没有通过修改 `sys.path` 解决导入问题？
27. 明确要求 Python 单元测试时，VS Code 测试视图是否能发现并运行测试，或是否运行了等价命令 `uv run python -m unittest discover -v -s ./tests -p "*_test.py" -t .`？
28. 新增或修改的源码文件和明确声明的单元测试是否都有文件级作用说明，并说明本文件维护的主要类、函数或测试对象？
29. 新增或修改的类、函数、方法和明确声明的测试方法是否都有职责说明、参数含义说明、返回值说明或明确无返回值说明？
30. 新增或修改的源码和明确声明的单元测试是否都遵守逐行意图注释标准，且每一行业务逻辑都有相邻说明注释？
31. Python 修改后是否运行了 Black 默认规范检查，例如 `python -m black --check src tests tools` 或项目等价命令？
32. Python 默认行为是否通过默认参数、配置对象、dataclass、factory 或小方法表达，而不是在 `__init__` 中堆叠大量分支？
33. 配置默认值是否直接写成默认参数，例如 `path=Config.default_config_file()` 或 `batch_size=config.train.batch_size`，并在函数体内用同名字段承载参数，例如 `self.batch_size = batch_size`？
34. Python 脚本是否没有新增命令行参数解析，且所有可调整行为都通过 config 子系统配置项表达，并且每个配置项都有默认值？
35. `src/<subsystem>/` 下的通用包是否有包内 `readme.md`，且 readme 是否只列公共接口和使用方式？
36. 代码实现是否逐条覆盖当前任务文档的预期行为，并且没有实现任务文档之外的功能？
37. 实现是否与 `00.用例.md` 的主成功场景和条件子步骤一致，尤其是成功路径、异常暴露方式和结束条件？
38. 实现是否与 `00.设计.md` 的复用入口、公共代码使用、设计决策和非目标一致？
39. 业务命名和对象职责是否表达领域含义，而不是只表达技术步骤或测试细节？
40. 是否遵守对应实现 skill 的最小实现纪律：绝对禁止臃肿代码，未提前框架化、泛化、抽象或增加需求外验证？
41. 是否没有捕获、吞掉、静默跳过、默认值兜底、空结果兜底、假装重试成功或包装任何会造成产品问题的异常；如存在处理，是否有明确业务要求、用户可见路径和验证？
42. 是否没有在大循环、训练循环、热路径或批处理内逐条检查数据合法性来兜底、过滤或拖慢性能？
43. 是否不存在多余校验、多余预检、多余中间变量、多余参数传递、多余重复变量名或参数名？
44. 是否不存在重复代码味道；如果存在相似逻辑、相似变量组、相似配置读取或相似转换，是否已经用 `$cx-common-module` 审视公共入口？
45. 代码是否保持极简、短小、直接，并且可复用功能、类或逻辑已经先使用 `$cx-common-module` 搜索和登记？
46. 是否已经清除所有源码、测试、脚本、工具、示例代码和流程生成代码中的臭长代码：没有把可由少量字段、直接数组切片、标准库语义或一个清晰构造函数完成的能力写成几百行甚至上千行；没有保留未被当前用例要求的协议继承、便利包装、复制方法、重建方法、补齐方法、负索引兼容、兜底校验、调试入口、未来扩展入口或测试专用入口？
47. 是否采用完整 OOP：Python 用类、dataclass、protocol 或显式对象协作表达状态和不变量；Rust 用 struct、enum、trait 和方法表达领域对象？
48. 是否避免了非 OOP 的动态访问方式，例如 `getattr`、`setattr`、`delattr`、monkey patch、动态注入或字符串分发；如不可避免，是否记录理由、隔离实现并验证？
49. 通用功能、可复用功能或可复用类是否先明确公共入口、常规调用方式、特殊场景入口、实例或状态生命周期、状态来源、验证如何覆盖所有源码调用和非目标？
50. 明确要求测试时，测试是否覆盖特殊入口、生命周期、状态恢复或测试隔离方式？
51. 是否存在只有一个调用点且没有隔离真实复杂度的 helper、包装函数、校验器或常量？
52. 是否存在需求未要求的验证、异常包装、动态构造、扫描器、注册表或未来扩展入口？
53. Rust 代码变更是否运行了必要 cargo 命令？
54. 文件、类、方法和变量命名是否短而清楚，避免过长描述性标识符；文件是否保持高内聚、不过度膨胀？
55. 各类交付物 review 是否都通过；如不通过，当前任务、变更或交付物是否仍保持未完成状态？
56. 文档交付物是否有明确读者、目标、范围、状态和唯一存放位置，并且没有散落文档、重复说明、过期说法或无依据结论？
57. 教程交付物是否能按顺序执行，前置条件、命令、输入、预期输出和失败处理是否明确，并且没有隐藏步骤或过时命令？
58. 研究交付物是否定义研究问题、时间窗口、纳入/排除标准、目标读者、来源质量和限制，并且每个非显然主张都有可靠引用？
59. 设计交付物是否说明目标行为、约束、不变量、公共入口、复用边界、非目标、取舍和可落地任务边界？
60. 工作是否满足提示词契约：目标、上下文、约束、必须遵循的流程、验证方式和交付物？
61. 最终摘要是否报告已运行命令、结果、各类 review 结论、跳过检查及原因、残留风险？
62. 对话摘要、cx 文档（本工作流维护的用例、任务、变更等文档）、审查结论和验证证据是否遵守语言规则；中文内容保留英文术语、缩写或外部专名时，是否在同句或相邻句说明中文含义、作用和当前上下文？

## 输出格式

```text
Findings:
1. [severity] file:line - issue
   Evidence:
   Fix:

Review decision:
- PASS 或 FAIL
- FAIL 时任务、变更或交付物状态必须保持未完成

Verified:
- command -> result

Missing evidence:
- ...
```
