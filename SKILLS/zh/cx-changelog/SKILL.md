# cx 变更文档维护

## 目的

维护当前主成功场景文件夹下的 `changes/`。变更文档是 AI 决定当前工作的第一入口：除非用户明确要求新增任务、新增条件子步骤或新增用例，否则 AI 必须先处理未完成变更。

## 规则

1. 每个主成功场景文件夹都必须有 `changes/`。
2. 变更文档只记录已经实施后的后续变化；未实施、未提交或初始规划阶段不创建变更文档。
3. 变更文档必须映射到当前 `00.用例.md`、`00.设计.md`、相关任务文档和一个生产代码文件；只有明确声明单元测试或 TDD 时才映射测试文件。
4. 如果一次变更涉及多个任务，或涉及用例文档修改，必须在变更文档中写出顺序任务清单。
5. AI 每次只执行顺序任务清单中的一项；完成后先做 `$cx-review`，交付前再用 `$cx-evidence` 核验证据，全部通过才允许更新为完成；默认停止在当前任务和当前代码文件边界，只有用户明确要求连续推进时才继续。
6. 不要创建第二份需求文档、草稿、旧版或并行计划。
7. 变更文档必须简洁但可审计：写清实施时间、关联对象、之前做了什么、现在应该如何、验证方式和处理结果。
8. release notes 只能从已完成变更、验证证据和通过的 review 结论汇总，不要把 release 内容提前写成另一份长期文档。

## 文件命名

```text
docs/cx/01.创建用户/changes/调整请求校验规则.md
```
