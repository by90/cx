# cx PyTorch 轻量调参

## 语言规则

- 使用中文输出对话、说明、计划、总结、审查结论、验证证据和 cx 文档（本工作流维护的用例、任务、变更等文档）时，必须使用完整中文表达，不得写成中英文混杂的半句或词组堆叠。
- 只有代码标识符、命令、路径、接口名、库名、协议、标准名、外部专名，或翻译后会造成歧义的术语可以保留英文；保留英文时，必须在同句或相邻句用中文详细解释其含义、作用和当前上下文。

## 资源安全前置门禁

任何轻量调参实验的第一步都是评估预计系统内存占用和显存占用，再决定是否启动。评估至少记录当前可用系统内存、当前可用显存、预计峰值、保留给操作系统和 Codex App 的安全余量，以及允许启动或必须调整窗口、批次或模型的结论；可以使用模型结构、张量形状、批量大小和最小试运行取得估算，但不得先启动完整实验再观察是否耗尽资源。十分之一完整实体样本是轻量调参的数据边界，资源不足时不得继续缩小样本比例；现有允许范围内的调整仍不能安全运行时，必须等待可用资源。

预计资源占用较多或耗时较长的任务必须在 Codex App 之外的外部终端启动，使训练进程和大量输出不由 Codex App 内置终端承载。训练进度、标准输出、标准错误、系统内存、进程内存、显存占用、每轮指标和错误必须持续写入文件型日志；人工智能代理只通过读取这些日志文件分析实验状态，不依赖终端滚动输出、终端控制序列或其它特殊终端处理。此门禁用于避免系统资源耗尽和系统崩溃，尤其避免 Codex App 因训练进程或终端输出负载而崩溃。

## 目的

轻量调参不是追求一次性得到最终最优模型，而是承担绝大多数参数选择：先确定数据 recipe，包括字段启用/禁用、字段分级数组、特征模式、标签定义和窗口长度；再确定模型结构和容量；同时选择 batch size、learning rate、optimizer、scheduler 等训练参数。轻量阶段最终保留 5 个候选，并在进入全量调参前对每个候选分别完成消融实验和回测。

## 硬规则

1. 训练、推理和回测入口必须能作为独立脚本运行，并在默认配置下完成一次有效执行；菜单或调参器只能调用这些脚本或它们背后的稳定入口。
2. 调参脚本必须先从目标模型默认配置完整初始化，再由公共调参设施通过 typed config hook 在内存中复制并修改配置，随后把配置对象传给训练、推理或回测入口；任何 trial 行为配置变更都只能作用于内存配置对象，包括字段、标签、窗口、batch size、learning rate、optimizer、scheduler、模型结构、输出目录和保存快照。不要为 trial 创建临时 YAML/临时配置目录作为事实源，不要直接改写当前配置文件作为 trial 行为，也不要在各个模型目录各自实现配置改写流程。保存 checkpoint 或权重时，必须把当次内存 recipe 对应的完整配置快照随权重一起保存。
3. Python 业务脚本、训练脚本、诊断脚本和一次性脚本不得用命令行参数改变行为；需要调整行为时，在 config 子系统定义带默认值的配置项。
4. 轻量调参固定使用完整实体样本总数的十分之一，抽样单位必须是完整实体。例如股票时间序列要抽股票而不是抽零散日期；总实体数不能整除 `10` 时记录取整方式、实际实体数量和实际比例。
5. Tensor 默认使用 `float32`；除非已说明理由，不要让数据管线或中间统计默默变成 `float64`。
6. 发现坏数据、越界分级、NaN、极端值或 shape 错误时，必须定位出错实体、日期、字段和值并解释原因；不要用过滤失败样本的方式掩盖问题。
7. 字段组合必须以“字段启用/禁用数组 + 显式字段顺序”表达。可以给候选 recipe 起可读名称，但持久化结果必须保存具体字段清单，不要只保存 `daily_fields` 这类模糊名称。
8. 字段分级必须以“每个字段选择几级 + 每级边界数组”表达。不同字段可以使用不同级数和不同边界；trial summary 必须保存最终展开后的 per-field bins。
9. 分级数组先由字段业务含义、理论最小/最大值、实际分布、中心区间宽度和极端值过滤规则确定；训练时优先搜索每个字段使用 7/9/11 还是一侧压缩后的级数，而不是临时乱改边界。
10. 有正负方向的相对指标通常使用 7/9/11 奇数级；只有单向非负或非正指标时，级数应压缩为一侧等级并保留“无/中性”含义。
11. 标签定义、标签类别选择和“最高/最低几级算机会或风险”的规则属于 series/数据层共享语义，不允许在单个模型层硬编码。
12. 特征模式要作为 recipe 维度研究：原始值、相对值、分级值、归一化值、排名值可以分别或分阶段比较；标签应保持同一口径，避免把特征模式变化误当成标签变化。
13. 排名更像横截面归一化阶段，不应默认永久混入所有基础字段。先验证基础字段、位置字段、排名变换、基础+排名是否各自有效，再决定是否保留排名字段。
14. 窗口长度（常见配置名为 `window_size` 或 `window_length`）和 batch size 必须绑定搜索。窗口越长，显存和计算量越高，尤其含 attention/pair 表征时可能接近随窗口平方增长；不要把短窗口的 4096/8192 batch 直接套到 20/40/60 长窗口。
15. 轻量调参必须覆盖短、中、长窗口。除非项目配置或显存明确不允许，窗口候选至少包含当前基准附近长度、`20`、`40` 和 `60`；长窗口要用显存预算缩小 batch 后验证。
16. 在双 NVIDIA RTX 5090、单卡 32GB 显存这类高显存环境中，轻量调参必须把显存利用率、GPU 占用率和训练吞吐作为一等约束。每个窗口/字段 recipe 都优先寻找最大稳定 batch：可行时从 `8192` 开始，不稳定或 OOM 时依次降到 `4096`、`2048`、`1024`、`512`；只有窗口过长、字段过多或模型结构导致爆显存时才逐级降低 batch，不要为了保守而长期使用小 batch 慢训。
17. batch size 和 learning rate 必须成对比较。batch 变大时可以略微增大学习率，batch 因长窗口变小时应重新验证学习率，不要孤立比较 batch 或 learning rate。
18. 分类任务默认以 `val_loss`/cross-entropy 作为 objective、调参和 early stopping 监控指标；`test_loss` 是泛化确认指标，回测是业务确认指标。
19. 当不同分级数量导致类别数不同，`val_loss` 的绝对值不一定可直接横比；同一标签/分级口径内先比 loss，跨口径时同时看 test_loss、改进次数、校准后的回测和错误分布。
20. 轻量训练上限固定为 `120` 个 epochs，early stopping 监控 `val_loss`，patience 固定为 `9`。每个 trial 必须记录是否跑满 `120` 个 epochs、`val_loss` 改进次数与改进比例、最佳轮次、结束轮次，以及最佳轮次距结束轮次的轮数和比例。
21. 多模型项目中，每个模型都可能有自己的最佳字段、分级、窗口和容量。保留模型内最佳候选，不要过早只保留一个跨模型全局排名。
22. 轻量调参必须建立五分钟跟踪节奏：启动后立即做一次基线采样，运行中每 5 分钟汇报当前 trial、当前 epoch、最近 `val_loss`、当前 best `val_loss`、`val_loss` 改进次数、当前 trial 已耗时、平均 epoch 耗时、预计剩余 trial/epoch、GPU 利用率、显存占用、显存峰值、系统内存和当前进程内存；如果 5 分钟内没有新 epoch 或新产物，必须说明停留位置、进程号和正在写入的资源日志路径。
23. 每个 epoch 和每个 trial 都必须写入统一资源监控产物。epoch 记录至少包含 epoch 序号、开始/结束时间、epoch 耗时、累计 trial 耗时、训练或验证 `val_loss`、当前 best `val_loss`、GPU 利用率、显存已分配/已保留/峰值/剩余、系统总内存/可用内存、进程内存、样本吞吐、是否 OOM 和错误原因。trial 结束记录至少包含 trial 总耗时、平均 epoch 耗时、完成 epoch 数、早停 epoch、`val_loss` 改进次数、best/final `val_loss`、`test_loss`、回测指标和资源日志路径。
24. 轻量调参候选汇报必须额外输出业务前十指标：Top10 命中率、命中候选平均涨幅、未命中候选平均涨幅、Top10 下跌占比和 Top10 下跌平均跌幅。五候选比较表必须同时包含 trial 名称、阶段、关键 recipe、`val_loss`、`test_loss`、训练 epoch、`val_loss` 改进次数、最佳轮次距结束轮次的轮数和比例、trial 耗时、平均 epoch 耗时和上述 Top10 指标。
25. 所有抽样、trial 初始化、模型初始化、数据打乱、消融实验和回测中的随机过程都必须读取 config 子系统中值为 `3407` 的种子配置。同一轮候选比较统一使用该配置值，不得在调参脚本中另写种子常量或为候选派生未记录的种子。
26. 轻量调参最终候选数量固定为 `5` 个。候选明确后，必须对每个候选分别执行消融实验和回测；消融时固定该候选之外的样本、种子和训练约定，逐项移除已选数据组、模型结构模块或关键训练组件，记录变化和结论。
27. 候选筛选首先比较训练可持续性：`val_loss` 改进次数和改进比例越充分、自然跑满 `120` 个 epochs、最佳轮次越接近结束轮次，信号越强；可持续性相近时，再比较 best/final `val_loss`、`test_loss` 和回测结果。

## 推荐流程

1. 先读项目 `AGENTS.md`、`docs/cx/00.项目说明.md`、当前主成功场景 `00.用例.md`、`00.设计.md`、相关 `tasks/` 和 `changes/`，以及已有实验摘要；重新开始调参时从当前仓库配置和已保存结果出发，不继承旧线程里的临时判断。
2. 检查是否有旧调参进程仍在运行；长时间 quick 必须按项目规则使用可见外部终端和临时防睡眠机制，并在结束、失败或交接前清理。
3. 启动或续跑前确认监控产物路径、五分钟采样方式和五候选汇报格式；如果项目没有统一资源监控器，先补公共监控入口和最窄测试，不要用终端刷屏替代结构化产物。
4. 用 `$cx-common-module` 搜索现有训练入口、推理入口、回测入口、config hook、recipe 持久化、资源监控和指标汇总能力；已有功能入口优先复用。
5. 用 `$cx-pytorch-tdd` 为配置映射、per-field bins、feature mode、窗口/batch 绑定、输出持久化、资源监控和指标解析补最窄测试；训练测试保持 CPU、小模型、小数据。
6. 做字段语义和相关性研究：记录每个字段的中文业务含义、理论最小/最大值、实际 min/max、分位数、中心区间、极端值规则、与标签的正负贡献和噪声嫌疑。
7. 生成字段候选时，用启用/禁用数组表达字段组合，至少分别验证：基础日线字段、基础日线+市场强度、基础日线+位置字段、基础日线+市场强度+位置字段、排名变换阶段、基础+排名组合。
8. 生成分级候选时，为每个字段准备 7/9/11 基准边界和必要的中心宽/窄变体；中心持平区间通常优先覆盖最宽分布区，两侧按业务值对称或近似对称。
9. 第一阶段固定模型参数和训练参数，只比较字段组合、字段分级数量/边界、特征模式和标签口径。该阶段不要同时搜索模型容量，避免把数据 recipe 和模型容量混淆。
10. 第二阶段固定数据 recipe，搜索窗口长度与 batch size。以当前最佳窗口为基准，同时覆盖 `20`、`40`、`60`；在双 RTX 5090 32GB 环境中，按 `8192`、`4096`、`2048`、`1024`、`512` 的顺序为每个窗口寻找最大稳定 batch，以不 OOM、显存利用率充分、GPU 占用率高和平均 epoch 秒数最低为准。长窗口、字段过多或模型结构导致 OOM 时才逐级下调，不要从低 batch 开始保守验证。
11. 第三阶段固定数据 recipe 和窗口/batch 预算，搜索 learning rate、weight decay、optimizer、scheduler、risk penalty 等训练超参；batch 增大时同步尝试略高 learning rate。
12. 第四阶段才搜索模型结构和容量，例如 hidden size、层数、heads、dropout、模板数、pattern 数、diffusion steps、encoder/decoder 深度。
13. 每个阶段至少用同一配置种子 `3407` 复跑当前 best recipe 一次，确认不是偶然随机结果；进入下一阶段前，把候选、失败 trial、pruned trial、五分钟跟踪摘要和排除理由写入当前 `docs/cx` story 的任务文档、变更文档或实验摘要。
14. 四个搜索阶段完成后固定 `5` 个候选。先按训练可持续性排序，再用 `val_loss`、`test_loss` 和业务指标消除接近方案之间的歧义；不得因为某次单一回测高而替换持续性更好的候选。
15. 对 `5` 个候选逐一执行消融实验和回测。消融实验固定十分之一完整实体样本、配置种子 `3407`、`120` 个 epochs 上限和早停忍耐 `9`；每个候选的基准结果、各消融结果和回测结果都必须保存，完成后把这 `5` 个候选交给全量调参。

## 评价口径

- 训练可持续性第一优先：综合 `val_loss` 改进次数、改进比例、是否跑满 `120` 个 epochs，以及最佳轮次距结束轮次的轮数和比例；最佳结果越靠近训练末尾，持续改善信号越强。
- `val_loss` 第二优先：决定训练目标和 early stopping；在可持续性相近时比较 best/final `val_loss`，不允许单次高回测覆盖明显更差的损失。
- `test_loss` 第三优先：用于判断验证集改善是否能泛化。
- 回测第四优先：用于业务验证；当训练可持续性、`val_loss` 和 `test_loss` 相近时，回测结果可以决定排序。
- 回测很高但 loss 明显更差时，优先怀疑偶然性、标签错位、分级边界、样本选择或收益统计口径。
- Top10 业务指标是股票候选类任务的强制汇报口径：命中率越高越好，命中均涨幅用于确认命中质量，未命中均涨幅越高越好，下跌占比越低越好，下跌平均跌幅绝对值越小越好。业务指标不能参与训练梯度或 early stopping，只用于候选解释、排序和晋级决策。

## 必须保存的证据

- 训练、推理和回测脚本在默认配置下可独立运行的命令和结果。
- 轻量调参抽样实体、严格十分之一的样本比例、不能整除时的取整方式、时间覆盖、数据版本、配置种子 `3407` 和硬件信息。
- 每个字段的业务含义、理论范围、实际分布、极端值处理、相关性和是否启用。
- 每个 trial 的字段启用数组、字段顺序、feature mode、per-field level count、展开后的 per-field bins、标签口径，以及与 checkpoint/权重一一关联的完整配置快照。
- 每个 trial 的窗口长度、batch size、峰值显存、显存余量、GPU 占用率、是否 OOM、样本吞吐和平均 epoch 秒数。
- 每 5 分钟跟踪摘要、采样时间、当前 trial、当前 epoch、最近 `val_loss`、best `val_loss`、`val_loss` 改进次数、trial 已耗时、平均 epoch 秒数、预计剩余量、GPU 利用率、显存占用、显存峰值、系统内存和进程内存。
- 每个 trial 的 optimizer、learning rate、weight decay、scheduler、模型结构参数和随机种子。
- `best_val_loss`、`final_val_loss`、`test_loss`、回测结果、最多 `120` 个训练 epochs、早停忍耐 `9`、早停 epoch、改进次数、改进比例、最佳轮次、结束轮次和二者距离。
- 五候选比较表，必须包含 Top10 命中率、命中均涨幅、未命中均涨幅、Top10 下跌占比、Top10 下跌平均跌幅、`val_loss`、`test_loss`、训练 epoch、`val_loss` 改进次数、最佳轮次距结束轮次的轮数和比例、trial 耗时和平均 epoch 耗时。
- `5` 个候选的复跑结果、逐候选消融矩阵、逐候选回测结果、进入全量调参理由，以及未进入五候选集合的排除理由。
