# Tutorial: Use Training Configuration

## Tutorial goal

Let training code construct `TrainingConfig` directly to read defaults and use constructor parameters for explicit current-run overrides.

## Audience and scenarios

- Developers writing training entries, model trainers, or data loaders.
- Maintainers adjusting learning rate, batch size, or other training parameters.

## Prerequisites

- The project Python environment was created with `uv`.
- The current task has implemented `TrainingConfig` in `src/config/training_config.py`.

## Public entry and source location

- Public entry: `src.config.training_config.TrainingConfig`
- Source location: `src/config/training_config.py`

## Inputs and outputs

| Item | Content |
| --- | --- |
| Input | Optional training-configuration constructor parameters |
| Output | A `TrainingConfig` instance containing current training parameters |

## Minimal runnable example

```python
from src.config.training_config import TrainingConfig

config = TrainingConfig()
print(config.batch_size)
```

Expected result: the configured default batch size is printed.

## Steps

1. Import `TrainingConfig` from the repository-root package.

   Expected result: Python resolves the public configuration entry.

2. Call `TrainingConfig()` directly to create default configuration.

   Expected result: the instance contains every default needed for training.

3. Read parameters from instance attributes in the training object.

   Expected result: the training entry parses no command-line arguments and maintains no second set of defaults.

## Common failures and handling

| Symptom | Cause | Correct handling |
| --- | --- | --- |
| Import fails | Import is not absolute from the repository root | Use `from src.config.training_config import TrainingConfig` |
| Two training defaults exist | The caller duplicates configuration values | Delete caller defaults and use the configuration object directly |
| A script requires command-line arguments | It bypasses the project configuration entry | Define the behavior as a configuration field with a default |

## Key constraints

- Training scripts do not accept command-line arguments.
- The configuration object owns defaults, and callers construct it directly.
- Do not retain compatibility for an old configuration entry.
