# AGENTS.md

## 启动必做：Windows 中文编码

在 Windows PowerShell 中读取、搜索、写入或展示中文文件前，必须先把当前命令通道设置为 `UTF-8`。`PowerShell` 是 Windows 命令外壳名称，`UTF-8` 是统一编码名称；本处保留原文是为了对应真实命令和编码标准。

在 Windows PowerShell 5.1 中先执行以下命令：

```powershell
$utf8NoBom = [System.Text.UTF8Encoding]::new($false)
[Console]::InputEncoding = $utf8NoBom
[Console]::OutputEncoding = $utf8NoBom
$OutputEncoding = $utf8NoBom
```

如果当前终端输出的中文出现乱码，禁止通过该输出判断文件内容是否正确；只能通过 `apply_patch`、结构化搜索、带明确编码的读取命令或已保存为 `UTF-8` 的脚本处理中文正文。发现乱码时只说明一次事实和处理方式；后续直接执行，不要在每次文件操作前重复输出同类提示。

## 启动必做：Windows PowerShell 5.1 命令兼容

在 Windows 原生 Codex App 中默认按 Windows PowerShell 5.1 编写命令，除非已经确认当前命令外壳是 PowerShell 7 或更新版本。`Codex App` 是本地桌面应用名称，`PowerShell 7` 是新版命令外壳名称；本处保留原文是为了对应真实产品和版本。

禁止使用 `&&` 和 `||` 这类 Windows PowerShell 5.1 不支持的命令连接符。需要连续执行时，使用多个独立命令调用；需要前一个命令成功后再执行后一个命令时，分步执行并检查退出结果。不要通过“先试一下失败再改”的方式确认命令外壳能力。

## 决策提示词

遇到架构取舍、接口设计、依赖选择、实现路径、性能与兼容性权衡、发布策略、反复缺陷或项目风险时，优先把问题套入以下 7 类提示词，让 agent 按明确框架追问、拆解并给出可执行结论。

1. **第一性原理思维**

   ```text
   我遇到一个工程问题，想用第一性原理重新审视。请不要直接沿用惯例、框架默认做法或常见实现套路；先把问题拆到最基础、可验证、不可继续拆分的事实和约束，再从这些事实出发重建解决方案。

   每一步都检查我的前提。如果我把某个假设当成事实，请指出它，并要求我补充证据或重新定义。

   问题：[详细描述技术问题、设计约束和目标行为]
   ```

2. **反向思维**

   ```text
   请用反向思维帮我分析这个工程目标。不要先问“怎样实现成功”，而是先问“怎样一定会失败”。

   请列出会导致实现失败、测试失败、维护失败或线上风险的行动、判断、假设和环境条件，再把每个失败因素反转成我现在应该采取或避免的具体动作。

   目标：[描述要实现的功能、修复、重构或发布结果]
   ```

3. **五问法**

   ```text
   我有一个反复出现的工程问题，需要找到根因，而不是只处理表面症状。请用五问法从我描述的问题开始，连续追问“为什么会这样”，至少深入 5 层。

   每一层都要比上一层更接近结构性原因。不要接受含糊回答；如果我的解释不清楚，请继续追问，直到找到真正可修复的根因。

   问题：[描述持续遇到的缺陷、构建失败、性能瓶颈、测试不稳定或维护问题]
   ```

4. **二阶思维**

   ```text
   我需要做一个工程决定，请用二阶思维帮我分析后果。不要只看直接结果，请分别梳理一阶结果、二阶结果和三阶结果，并同时分析“做”和“不做”两条路径。

   决定：[描述当前架构、接口、依赖、数据模型、迁移、排期或发布决策]
   ```

5. **后悔最小化框架**

   ```text
   我被一个重要工程选择困住了。请用长期维护视角的后悔最小化框架帮我思考：假设 6 到 12 个月后团队继续维护这段代码、接口或架构，会如何评价今天的选择？

   请推动我诚实面对维护成本、测试成本、扩展边界和团队认知负担。

   工程抉择：[描述面临的技术方案、架构路径或实现选择]
   ```

6. **机会成本分析**

   ```text
   请用机会成本分析帮我评估这个工程承诺。每次我对一个功能、重构、依赖、工具或发布动作说“是”，也意味着对其他可用同样时间、精力或风险预算完成的工程工作说“不”。

   请把问题框定为“这个选项对比那个选项”，而不是简单的是或否。

   工程承诺：[描述正在考虑的功能、重构、依赖引入、工具建设或发布动作]
   ```

7. **事前验尸分析**

   ```text
   我准备启动一个开发项目、重构、迁移或发布，请先做事前验尸分析。假设 6 个月后它已经彻底失败，请倒推列出失败的所有可能原因。

   请给出具体工程场景，说明哪里出错、什么时候出错、为什么测试、评审、监控或发布流程当时没有发现，并为每个失败场景给出一个现在就能采取的预防动作。

   工程项目：[描述即将启动的功能、重构、迁移、基础设施工作或发布]
   ```

## 铁律

这些铁律优先级高于本文件其他规则、所有 cx skills、模板、示例和临时任务说明；只有用户在当前对话中明确覆盖某条铁律时，才允许按用户的新指令执行。

1. **提交和 push 不分辨来源**：用户要求提交、交付、发 PR 或发布时，把当前工作区作为一次完整变更处理，暂存并提交已跟踪和未跟踪文件；不要分析哪些文件是谁改的，也不要按来源拆分提交。
2. **全面注释**：所有新增或修改的源码和单元测试都必须有代码文件级说明、类说明、函数说明和逐行业务代码说明；文件级说明必须说明本文件的作用和主要类、函数或测试对象，函数或方法必须解释参数含义和返回值或明确无返回值；只有空行、纯格式行或重复结构行可以不写相邻说明注释。
3. **默认单任务单代码文件推进**：新 story 建立时一次性确定任务集合和每个任务绑定的生产代码文件。开始编程或流程性研发工作前，默认先完成当前原任务文档，再只编辑该任务绑定的一个生产代码文件。story 建立后需要改变文件或实现方式时，先登记临时变更并修改原任务，不得新增、删除或重命名任务文件。只有用户明确要求连续推进多个原任务时，才继续处理下一个代码文件；只有用户明确要求逐任务确认时，才在每个原任务后等待核对。
4. **中文包全中文**：安装中文 cx 包时，所有 cx 生成或维护的文档必须使用简体中文；代码标识符、命令、接口名、库名和外部专名可以保留原文。
5. **单元测试显式声明**：默认不创建、不修改、不运行单元测试；只有用户当前请求、既有任务文档或变更文档明确声明需要单元测试、TDD、失败测试或 red-green-refactor 时，才创建或修改一个对应测试文件。需要单元测试时，测试目录必须镜像 `src` 结构，源文件 `src/<subsystem>/xx.py` 只能对应 `tests/<subsystem>/xx_test.py`；不要用一个大测试文件覆盖多个源文件，也不要为同一源文件拆出多个随意命名测试文件。
6. **默认参数优先**：构造函数和普通函数优先使用清晰的类型标注与默认参数表达默认行为；配置默认值应直接写成默认参数，例如 `path=Config.default_config_file()` 或 `batch_size=config.train.batch_size`，函数体内用同名字段承载参数，例如 `self.batch_size = batch_size`；不要在 `__init__` 里堆叠大量 `None` 分支、重复命名或参数情况判断，复杂默认构造应放到 dataclass、配置对象、factory 或小型专用方法中。
7. **完整面向对象、极简和复用优先**：默认使用完整面向对象设计表达状态、生命周期、不变量和领域协作；禁止随意写出臃肿、冗长、难维护的代码；避免过长文件、过长变量名和句子式标识符。任何可能复用的功能、类或逻辑都必须先使用 `$cx-common-module` 搜索和设计功能入口。绝对禁止默认使用 `getattr`、`setattr`、`delattr`、monkey patch、动态注入或字符串分发；只有没有静态面向对象接口可走时才允许，并且必须记录理由、隔离实现和验证。
8. **Python 脚本不接收命令行参数**：目标项目中新增或修改的 Python 业务脚本、训练脚本、诊断脚本、迁移脚本和一次性脚本都禁止使用命令行参数改变行为；需要调整行为时，必须在 config 子系统定义对应配置项，并为每个配置项提供清晰默认值。
9. **开发阶段只保留最新意图**：项目开发阶段所有代码、接口、参数、配置、路径、调用方、文档、示例和明确声明的测试都只针对当前最新意图。绝对禁止保留旧接口、旧入口、别名、适配层、桥接函数、新旧并存分支、兼容参数、兼容配置、兼容路径、旧行为兜底或任何旧实现痕迹。调用方必须同步迁移到当前入口；修改完成后删除全部旧代码、旧文档、旧示例和旧测试痕迹。
10. **异常原样抛出铁律**：除非用户在当前请求中明确要求某项具体校验或异常处理，否则编程实现绝对禁止自行增加一系列校验后主动抛出异常，也绝对禁止捕获、转换、包装、吞掉、静默跳过、默认值兜底、空结果兜底、假装重试成功或降级处理异常。底层异常必须保持原始类型、原始信息和原始调用栈，自然抛出并中止程序。正常业务状态可以用明确对象或返回值建模，但程序异常不得转成业务分支继续运行。
11. **性能路径不做兜底式预检**：严禁用大量预先检查、大循环内逐条合法性检查、训练循环检查、热路径兜底或批处理静默过滤来掩盖数据错误或拖慢性能；数据合法性应在入口、数据准备、测试夹具或单独诊断任务中处理。
12. **非编程不自动创建用例流程**：普通、非编程任务不要自行创建 cx 用例、任务或变更文档；如果不确定该任务是否需要行为发现或验收场景，先向用户提出一个最小澄清问题。
13. **可见待办事项**：开始任何需要多步执行的工作时，必须先在对话中创建待办事项列表；执行过程中逐项更新状态，只有所有事项完成、取消或明确阻塞并说明原因后，才能结束本轮工作。
14. **Python 脚本使用 uv Python**：运行 Python 脚本、测试、构建或工具命令时，必须使用 `uv` 安装或同步的 Python；系统自带 `python`、`python3` 只能用于确认环境，不能代替项目 Python。
15. **中文表达不夹英文**：人工智能代理在对话、说明、计划、总结、文档、审查结论、验证证据和注释中使用中文时，必须使用完整中文表达，不得把中文句子写成中英文混杂。只有代码标识符、命令、路径、接口名、库名、协议、标准名、外部专名，或翻译后会造成歧义的常见英文可以保留；保留英文时，必须在同句或相邻句用中文详细解释其中文含义、业务作用和本处为什么保留英文。
16. **跨平台编码铁律**：PowerShell 脚本 `*.ps1`、`*.psm1`、`*.psd1` 必须保存为带 BOM 的 UTF-8；除 PowerShell 脚本外，Python、Shell、Markdown、JSON、TOML、YAML、文本和其他源码或配置文件必须保存为 UTF-8 无 BOM。Windows 个人电脑可以正常处理中文项目，但必须区分文件编码、控制台显示编码、管道编码、外部程序参数编码和终端字体；不得把其中一个环节的乱码误判为文件损坏。Windows 上优先使用 PowerShell 7 或更新版本处理非 PowerShell 文本，因为它默认使用 UTF-8 无 BOM；使用 Windows PowerShell 5.1 时，必须显式设置 `[Console]::InputEncoding`、`[Console]::OutputEncoding`、`$OutputEncoding`，并为 `Get-Content`、`Set-Content`、`Select-String`、`Out-File` 等文本命令传入明确编码。允许用脚本批量修改中文内容，但脚本文件、输入文件、输出文件和调用通道都必须明确使用 UTF-8，并且必须保持目标文件原有 BOM 规则；不得把中文替换表、中文正文或中文代码通过未验证编码的 PowerShell 内联字符串、命令行参数或 here-string 传给外部程序。需要批量处理时，优先使用已保存为 UTF-8 的脚本文件、项目 `uv` Python、PowerShell 7 的 `-Encoding utf8NoBOM`，或明确无 BOM 的 .NET 写入方式；需要精确小改时，使用 `apply_patch`。只有发现文件实际编码异常、补丁失败或用户主动要求处理编码时，才允许修改文件编码或说明编码原因。
17. **先中文后英文**：修改 cx 流程、技能、模板、示例或安装规则时，必须先完成中文包，再把同等结构和约束同步到英文包；如果只能选一侧，先改中文。
18. **显式测试和完整面向对象**：Python、PyTorch 和 Rust 项目默认不使用单元测试或 TDD；只有明确声明时才执行测试先行。涉及状态、生命周期、不变量和领域协作时必须使用完整面向对象或等价类型建模。
19. **交付物完成后强制 review**：代码、文档、教程、研究、设计、流程变更或发布说明等任何交付物完成后，必须用 `$cx-review` 连续执行交付物质量审查和完成证据门禁，再认定任务或交付物完成。代码审查必须核对实现是否与当前 `00.用例.md`、`00.设计.md`、原任务文档和未完成变更文件完全一致，是否先读取专题文档并复用已登记通用能力，是否存在重复味道，是否完整面向对象，是否极简且没有多余校验、多余变量传递、多余重复变量或参数命名，以及业务含义是否吻合。文档、教程、研究和设计审查分别核对当前状态唯一性、教程可执行性、来源质量与综合结论、设计可落地性和业务含义。任一阶段不通过时任务或交付物仍未完成，当前变更文件不得删除。
20. **臭长代码必删铁律**：任何源码、测试、脚本、工具、示例代码或流程生成的代码，若把本可由少量字段、直接数组切片、标准库语义或一个清晰构造函数表达的能力写成几百行甚至上千行，必须视为未完成和 review 不通过。这是跨语言、跨目录、跨 skill 的铁律，不得因为 TDD、兼容、调试、未来扩展或“看起来完整”而放宽。禁止添加未被当前用例要求的协议继承、便利包装、复制方法、重建方法、兜底校验、负索引兼容、掩码补齐、调试入口、未来扩展入口或测试专用入口。写代码前必须先列出最小功能入口和非目标；写完后必须主动删除没有真实调用点、没有减少调用复杂度、没有隔离真实复杂度的函数、属性、类和测试。代码应该直接、短小、贴近数据结构和业务语义；能让调用方直接使用底层数组、张量、标准切片、构造函数或配置默认值表达的，不再包一层。
21. **全局数值类型铁律**：所有新增或修改的 `NumPy` 数组、PyTorch 张量、模型输入、指标矩阵、中间结果、训练数据、分级值、类别值、实体序号、时间序号、行号、分组偏移和索引数组都必须使用项目配置中的类型对象。项目应在 config 子系统中定义连续数值类型、分级类别类型和索引类型的配置项，并让性能路径直接使用类型对象，不在循环、数据读取、指标计算、组批、训练、推理、损失计算或测试夹具中解析字符串。除配置定义、配置默认值测试断言和配置文档说明默认值外，禁止在生产路径、测试夹具和文档示例中直接写 `np.float32`、`np.float64`、`np.float16`、`np.int64`、`np.int32`、`np.int8`、`torch.float32`、`torch.float64`、`torch.float16`、`torch.bfloat16`、`torch.int64`、`torch.int32` 或等价硬编码类型。当前项目如果使用 `NumPy` 数据层且 `NumPy` 没有原生 `bfloat16`，不得用第三方库扩展 `NumPy` 来提供该类型；机器学习侧可以通过 PyTorch 模型配置选择 `torch.bfloat16`。
22. **cx 源仓库修改铁律**：修改 cx 流程、技能、模板、示例、安装规则或全局 `AGENTS.md` 模板时，只能修改 cx 源仓库，不得直接手改本机全局安装目录作为最终结果。每次修改后必须提交并 push 到 `main`，随后运行对应安装脚本从远端更新本机全局 cx 技能和 `AGENTS.md`。
23. **当前状态与 Git 历史铁律**：项目说明、用例、设计、任务、专题文档、研究笔记和接口说明只允许表达当前有效状态，绝不记录旧方案、新旧差异、迁移过程、完成变更或开发过程。新旧主要差异只允许存在于当前未完成的 `changes/` 文件中。变更文件必须先提交到 Git，再开始修改正式文档和实现；统一审查通过后必须删除变更文件并再次提交。Git 提交、标签和发布记录是唯一历史来源。
24. **项目 AGENTS.md 铁律**：除全局 `AGENTS.md` 外，每个项目根目录都必须有根据项目目标、主要编程语言、工具链、目录结构、运行入口、验证命令和通用包填写的项目 `AGENTS.md`。项目文件只补充项目事实，不复制、削弱或覆盖全局规则。初始化或修正项目规则时使用 `$cx-doc` 的 `assets/AGENTS.md` 模板。
25. **通用包教程铁律**：每个通用包、可复用组件和稳定公共接口都必须在 `docs/cx/docs/` 下拥有站在调用者角度的独立编号教程。项目 `AGENTS.md` 必须登记所属领域、教程链接、公共入口和必须先读的场景。进入相关领域前必须先读全部链接教程；缺少教程或导航时，先使用 `$cx-doc` 补齐，不得直接规划、设计或编码。
26. **绝对禁止顺手处理任务外问题**：编码、测试、重构、审查、文档和配置修改只能处理用户当前请求、当前用例路径、原任务文档和未完成变更共同界定的当前任务。发现与当前任务无关的问题时，无论问题看起来多明显、多简单、多严重，或代理认为多值得按最佳实践修复，都绝对禁止顺手修改、删除、格式化、重构、补测试、补文档或调整配置。只能把问题列入待授权清单，逐项写明文件、位置、现象、风险和建议，并请求用户对具体清单项明确授权；没有取得该项授权前不得处理。代理自己的推测、偏好、审查意见和“为了代码更好”的判断都不构成授权，也不得扩大任务范围。

## 仓库工作约定

本仓库使用 `docs/cx` 用例驱动流程。所有项目说明、场景、任务、流程和变更文档均在 `docs/cx` 下；其它任何地方的文档与 cx 无关。

1. cx 文档必须写具体事实、具体动作和具体决策；一句话只表达一个意思，禁止空泛铺垫、重复目标、遗漏“要做什么”和自造未定义术语。
2. 项目说明只写项目目标、关键术语、设计约束和用例索引；目标用 `1. 2. 3.` 描述，每项目标必须说明要达到什么结果以及为什么需要该结果。
3. 用例只写参与者、前置条件、触发条件、主成功场景、挂到具体主步骤的条件子步骤、子用例和完成结果；不要在用例正文里写测试计划、实现任务或重复项目目标。
4. 设计文档只写文件范围、功能入口、可复用能力、设计决策、取舍理由和验证方式；任务文档只写本任务要做什么、文件范围、任务量具、验证方式和状态。
5. `changes/` 只保存当前尚未完成的工作指令；已有 story 发生需求变化、设计改向、实现改向或代码错误时，先创建或更新变更文件并提交到 Git。
6. 新术语首次出现必须定义；优先使用用户和项目原词，禁止用团队未定义词替代明确对象。
7. 规划或修改代码前，先读取项目根目录 `AGENTS.md`，再按其中的领域导航阅读全部相关通用包教程，然后阅读 `docs/cx/00.项目说明.md`、目标主成功场景的 `00.用例.md`、`00.设计.md`、原任务和未完成变更。
8. AI 开始工作时先按项目 `AGENTS.md` 搜索已经登记的通用包、教程、公共入口和真实调用点，再查看未完成变更，并以未完成变更决定当前应该做什么。
9. 每个主成功场景一个文件夹，例如 `docs/cx/01.创建用户/`。
10. 每个主成功场景文件夹必须包含 `00.用例.md`、`00.设计.md`、`tasks/` 和 `changes/`。
11. `00.用例.md` 统一表达主成功场景，以及挂靠到具体主步骤的条件、替代和异常子步骤；复用代码场景和业务场景都使用同一种表达方式。
12. 一个 `docs/cx/NN.名称/` 文件夹只能承载一个用户目标用例，不要用一个“使用应用”类用例容纳整套应用、页面集合或多个互相独立的操作。
13. 主成功场景只写该用户目标从触发到目标达成的最常见成功路径，通常 3 到 9 个主步骤；每一步必须是参与者和系统之间的一个可观察交互。
14. 如果主成功场景某一步本身需要参与者、前置条件、主成功场景和条件子步骤，说明它已经是子用例或独立用例，应拆成新的 `docs/cx/NN.名称/` 文件夹，并在原步骤下用子编号写明“进入用例：某某”。
15. 条件、替代和异常必须用 `1.1`、`2.1` 这类子编号挂到具体主成功场景步骤；复杂条件流程如果超过 3 到 5 步、引入新参与者、独立完成标准或独立任务测试，应拆成单独用例。
16. `00.设计.md` 说明可使用哪些公用代码、需要编写哪些公用代码，以及关键设计决策。
17. 每个任务一个 Markdown 文件，从 `01.` 开始用序号加中文名命名，例如 `tasks/01.编写用户实体.md`；不要创建 `00.任务.md` 这类泛名文件。
18. 每个变更一个无时间戳 Markdown 文件，用中文变更名命名，例如 `changes/调整用户实体约束.md`。
19. 当一次变更涉及多个原任务或用例文档修改时，必须在变更文件中列出顺序工作清单；默认每次只完成当前原任务和一个生产代码文件，用户明确要求连续推进时才进入下一项。
20. 新 story 建立时一次性确定任务集合；story 建立后任务文件数量、编号、文件名和身份不得改变。实现改向、需求调整或代码错误直接修改原任务文档，不创建“修复”“移除”“修改”或“再次实现”任务。一个任务只处理一份任务文档和一个生产代码文件；只有明确声明单元测试或 TDD 时，才额外处理一个一一对应的单元测试文件。
21. 任务文档的基本量具是类或类型组合，可以是一个类，也可以是一组紧密协作的类或类型。
22. 流程处理、任务分流和不确定应使用哪个 cx skill 时，优先使用 `$cx-workflow`。
23. 用例、主成功场景、条件子步骤、固定任务集合和当前状态维护使用 `$cx-story`；临时变更文件使用 `$cx-changelog`。
24. 明确声明 TDD 或单元测试时先使用 `$cx-tdd` 执行唯一主流程；Python、PyTorch 或 Lightning 测试再叠加 `$cx-pytorch-tdd`，Rust 测试再叠加 `$cx-rust-tdd`。没有明确测试要求时不使用两个语言测试技能。
25. 通用功能、可复用功能或可复用类实现前，必须先使用 `$cx-common-module` 搜索和设计功能入口。
26. 修改后先运行最窄的有效测试，再按需要运行更宽的验证，并记录命令和结果。
27. 通用包和稳定公共接口必须使用 `$cx-doc` 在 `docs/cx/docs/` 下维护独立编号的调用者教程；协议、数据过程、特征体系和技术转向维护独立编号专题文档。项目 `AGENTS.md` 必须链接所有通用包教程；工作开始时先读取当前领域教程，再搜索源码和真实调用点。
28. 任一研究必须在 `docs/cx/notes/` 下保存针对具体问题的当前结论；结论先回答问题，再用流畅、通俗的中文说明依据、限制和工作影响。
29. 任一交付物完成后必须使用 `$cx-review` 连续执行交付物质量审查和完成证据门禁。任一阶段不通过时任务仍未完成，当前变更文件不得删除。

- `$cx-research`：模型选择、AI 论文研究、来源筛选和带引用综合分析。
- `$cx-design`：面向对象设计、职责拆分、领域对象、类命名、继承组合、数据库访问边界、字段枚举和实现路径取舍。
- `$cx-pytorch-tdd`：在 `$cx-tdd` 主流程上补充 Python、PyTorch 和 Lightning 的测试工具、布局、真实数据与张量检查。
- `$cx-pytorch-quick-hpo`：PyTorch 轻量调参使用十分之一完整实体样本，选择数据参数、模型结构参数和训练参数，并对 5 个候选分别执行消融实验和回测。
- `$cx-pytorch-full-hpo`：PyTorch 全量调参使用全部样本，保持轻量阶段的数据与模型参数不变，只调整批次大小、学习率、优化器参数和调度器参数，并逐一训练、测试和回测 5 个候选。
- `$cx-timeseries-modeling`：异构多变量时间序列建模、字段语义分层、协变量、泄漏检查和 PyTorch Forecasting 选型。
- `$cx-rust-tdd`：在 `$cx-tdd` 主流程上补充 Rust 内置测试、共享真实数据夹具和 `cargo` 检查。
- `$cx-common-module`：通用功能、可复用功能、可复用类、可复用能力抽取和功能入口设计。
- `$cx-review`：代码、文档、教程、研究、设计和流程变更完成后的强制本地 review。

## Python 规则

- 子系统源码必须放在 `src/<subsystem>/` 下；例如 config 子系统放在 `src/config/`，其中 CNN 配置应是独立文件 `src/config/cnn_config.py`。
- 使用项目级 `uv` 虚拟环境；安装依赖和运行 Python 命令优先使用 `uv sync`、`uv run` 或项目已有的 `uv` 工作流。
- Python 解释器必须优先使用 `uv` 安装和管理的版本；系统自带 `python` / `python3` 只能用于确认环境。
- 创建或重建 Python / PyTorch 环境前，访问 Python 官网和 PyTorch 官网，确认 Python、PyTorch 与 CUDA 组合为当前官方稳定版本；不要默认使用 nightly、预发布或非官方轮子。
- `NumPy`、PyTorch、指标矩阵、训练数据、分级值、类别值、序号、行号和索引数组必须使用 config 子系统中的类型对象；除配置定义、配置默认值测试断言和配置文档说明默认值外，禁止在业务代码、性能路径、测试夹具或文档示例中直接写硬编码数值类型。
- 默认使用完整面向对象组织领域逻辑；纯粹无状态且不可复用的小逻辑可以保留为短函数，对状态、生命周期、不变量和领域对象协作必须使用面向对象设计。
- 构造函数和普通函数优先用类型标注与默认参数表达默认行为；配置默认值应直接写成默认参数，例如 `path=Config.default_config_file()` 或 `batch_size=config.train.batch_size`，函数体内用同名字段承载参数，例如 `self.batch_size = batch_size`；不要在 `__init__` 里用大量 `if`/`None`/类型分支处理各种参数组合。
- 通用包必须在 `docs/cx/docs/` 下提供独立编号的调用者教程，说明目标、场景、前置条件、公共入口、输入输出、最小可运行示例、步骤、预期结果、常见失败、约束和验证方式；其它文档只链接该教程，不复制正文。
- 目标项目中的 Python 脚本不得通过 `argparse`、`click`、`typer`、`sys.argv` 或自定义参数解析接收命令行参数；脚本行为、路径、设备、训练参数、模式开关和诊断开关必须来自 config 子系统。
- config 子系统中为脚本行为定义的每个配置项都必须有默认值。
- 禁止默认使用 `getattr`、`setattr`、`delattr`、monkey patch、动态注入方法或字符串分发；只有没有明确静态面向对象接口时才允许，并且必须记录理由、隔离实现和测试。
- 代码必须保持短小、直接、可读，避免过长文件、过长变量名和句子式标识符；发现可复用功能、类或逻辑时，先使用 `$cx-common-module` 搜索已有实现和可复用能力登记。
- 代码格式遵循 Black 默认规范；修改 Python 源码或测试后必须运行 `python -m black --check src tests tools` 或项目等价命令。
- 明确声明 Python 单元测试时，使用 Python 自带的 `unittest`，禁止引入 `pytest`；只有用户当前请求明确要求时才例外。
- 明确声明单元测试时，单元测试目录必须镜像 `src` 目录结构，并与源代码文件一一对应：`src/config/cnn_config.py` 的测试文件必须是 `tests/config/cnn_config_test.py`。
- `src/` 及其子目录必须有空白 `__init__.py`。`tests/__init__.py` 必须集中读取测试数据库并构造共享真实数据对象，不得留空；`tests/` 的其它包目录保留空白 `__init__.py`。
- 项目内导入必须使用从仓库根包开始的绝对导入，例如 `from src.config.config import Config`；测试不得通过修改 `sys.path` 解决导入问题。
- 明确声明单元测试时，VS Code 测试视图必须能发现并运行测试；`unittest` 发现参数使用 `-v -s ./tests -p *_test.py -t .`。
- PyTorch 和 Lightning 接口或版本相关行为必须查官方最新文档。
- tensor 测试要覆盖 shape、dtype、device、确定性和边界场景。
- 训练测试必须很小：优先 CPU、小 batch、小模型、`fast_dev_run` 或有限 batch。
- 明确声明数据相关单元测试时，必须使用测试数据库真实数据；`tests/__init__.py` 只读取一次并提供共享对象，各测试文件不得重复读取。除非用户当前请求明确要求，禁止 mock 测试。

## Windows 工具链规则

- 在 Windows 上，如果任务明确需要 `ng` 命令但本机没有安装，先安装项目要求的 `ng` CLI，再继续执行。
- 在 Windows 原生 Codex App 中，默认使用 PowerShell 运行命令和本地设置脚本；只有代理切换到 WSL 或项目明确位于 WSL 环境时，才使用 WSL 命令环境。
- 优先使用项目锁定的包管理器安装 `ng`，例如已有 `package-lock.json` 时使用 `npm`，已有 `pnpm-lock.yaml` 时使用 `pnpm`，已有 `yarn.lock` 时使用 `yarn`。
- 安装后运行 `ng version` 或项目约定的等效版本命令，确认 CLI 可用，再运行后续生成、构建或测试命令。

## Rust / GPUI 规则

- 明确声明 Rust 单元测试时，使用语言内置测试机制和 `cargo test`，禁止引入额外测试框架；只有用户当前请求明确要求时才例外。
- Rust 修改后运行 `cargo fmt`；只有明确声明单元测试或 TDD 时才运行 `cargo test`，可行时运行 `cargo clippy --all-targets --all-features`。
- Rust 数据相关测试必须通过统一测试夹具模块读取测试数据库真实数据，并使用一次初始化机制共享；各测试模块不得重复读取数据库。除非用户当前请求明确要求，禁止 mock 或伪造仓库测试。
- 用 struct/enum/trait 和明确 `Result` 错误表达领域状态。
- 生产路径避免 `unwrap`、`expect` 和 `panic!`，除非不变量局部、已证明并记录。
- 将纯状态、reducer 和渲染代码分离。
- 新增可复用 UI state、组件接口或 reducer 前，先搜索可复用能力登记和已有实现。
- 尽可能使用无状态 gpui-component 元素，由 view 持有状态。
- UI 组件接口保持小而可复用。
- 修改 Rust/GPUI 桌面 UI 后，必须打包、安装或启动真实应用进行实机观察；不能只依赖静态检查、明确声明的单元测试或想象中的截图宣布 UI 完成。
- macOS 辅助功能授权必须覆盖发起自动化的宿主和被测应用。
- GPUI 内容区控件通常不会完整暴露为 macOS Accessibility 树中的按钮；`System Events` 优先用于菜单栏、窗口按钮和页面菜单动作。
- 点击 GPUI 内容区前先把截图保存到项目 `temp/`，确认显示器 bounds、窗口位置、窗口尺寸和 Retina 缩放。
- 临时截图、坐标标定图、安装包和 UI 验证产物必须放入项目 `temp/` 或项目约定临时目录，并确保不会进入源码库。

## Git 提交规则

- 用户要求提交、交付、发 PR 或发布时，把当前工作区视为一次整体变更，不要分析哪些文件是自己修改的、哪些是用户修改的、哪些是尚未跟踪的。
- 提交前运行 `git status --short` 只用于确认工作区内容和发现明显风险，不用于按来源拆分文件。
- 默认暂存所有已跟踪和未跟踪文件，并创建一个提交；不要把同一任务拆成多个提交，除非用户明确要求。
- 只有发现明显密钥、凭据、本地环境文件、构建产物、依赖目录或无关大文件时，才先停止并请求用户确认。
- 提交信息应描述用户请求对应的整体结果，而不是逐文件列出来源。
- 工作完成并经过用户确认后，把本地工作分支合并到 `main`，随后删除本地工作分支。
- 除非用户在当前对话中明确覆盖 main-only 远端策略，否则不要把工作分支推送到远端。
- 默认只推送 `main` 和版本 tag。

## 运行环境与 UI 检查

- 开始长时间运行、构建、测试、安装或 UI 实机检查前，先阅读项目 `AGENTS.md`、README 或脚本目录，查找适用于当前平台的防睡眠、防锁屏或保持会话活跃机制。
- 防睡眠机制必须是临时的、可恢复的，并在本轮工作结束、阻塞或交接前关闭。
- macOS 桌面或 GUI 项目完成 UI 相关修改后，必须先按项目方式打包、安装或启动真实应用，再使用 Computer Use 或项目指定的实机检查方式实际运行并观察结果。
- UI 实机检查、安装或打包命令、观察结果和剩余风险必须写入当前任务文档；未完成变更文件只维护当前工作指令。没有目标 cx 文档时，必须在最终摘要中说明。
