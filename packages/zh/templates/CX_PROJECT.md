## 技术约束

- Python、PyTorch 和 Rust 工作默认不使用单元测试或 TDD；只有用户请求、任务文档或变更文档明确声明时才创建或修改单元测试。
- 涉及状态、生命周期、不变量和领域协作时，必须使用完整面向对象或等价类型建模。
- 新 story 建立时一次性确定任务集合；已有 story 的任务文件数量、编号、文件名和身份不得改变。
- 默认完成一个任务文档后只编辑一个生产代码文件；实现改向或代码错误直接修改原任务，不创建修复、移除或修改任务。
- 任务文件使用 `tasks/NN.中文任务名.md`，不要使用 `00.任务.md`。
- 变更文件使用 `changes/中文变更名.md`，只指导当前未完成工作；文件先提交到 Git，完成审查后删除并再次提交。
- 代码必须极简、复用优先、低重复，避免过长文件、过长变量名和句子式标识符。
- 通用包和稳定技术过程必须在 `docs/cx/docs/` 下提供独立编号专题文档；工作开始时先读取已有专题文档。
- 每项研究都在 `docs/cx/notes/` 下保存针对具体问题的当前结论和通俗说明。
- 所有 cx 场景、任务、流程和变更文档只能放在 `docs/cx` 下。
- 所有正式文档只表达当前有效状态，不记录旧方案、新旧差异或完成变更；历史由 Git 保存。
- 代码、文档、教程、研究、设计或流程变更等交付物必须通过 `$cx-review` 的质量审查和完成证据门禁。
