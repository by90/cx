# BDD/TDD AI 辅助开发流程

## 核心思想

BDD 定义行为和共同语言。TDD 用可执行测试证明行为。AI 编程 agent 只有在被长期文档和验证证据约束时，才真正可靠。

cx 工作流采用这个顺序：

1. 接收用户需求。
2. 修改项目文件前，先创建或切换到该功能组的独立分支。
3. 选择或创建目标编号功能文件夹，例如 `docs/1.配置系统/`。
4. 在目标 `BDD.md` 中更新业务规则、主成功场景、分支场景和异常场景。
5. 在目标 `CHANGELOG.md` 中更新 `CHANGE-*` 条目。
6. 在目标 `ENGINEERING_SPEC.md` 中更新架构说明、任务队列和回链 BDD ID 的测试矩阵。
7. 停止并向用户汇报文档结果和下一步实现计划，等待用户明确确认。
8. 用户确认后，编写预期失败的测试。
9. 运行最窄测试并记录 red failure。
10. 实现最小改动。
11. 运行测试直到 green。
12. 只在测试保持 green 的情况下重构。
13. 记录验证证据。
14. 搜索已有实现、相关 skills 和 registry，判断重复逻辑是否应该成为可复用功能、类或组件。
15. 功能组完成后，把功能分支合并到 `dev`；从 `dev` 交接到 `main` 的发布流程必须等待用户确认版本完成。

## 为什么坚持 docs 文档集

很多 AI 工作流会为每个需求生成新的 spec、plan 或 task 文件。单看一次任务似乎清楚，但长期项目会变成无法搜索的文档垃圾。cx 让所有新工作进入明确的 `docs/` 文档集：小项目可以只有一个文档集，大项目按功能组拆成多个目录，并用 `docs/INDEX.md` 统一索引。

## 推荐提示词

```text
请使用 $cx-bdd 和 $cx-tdd。先创建或切换到功能组分支，再选择或创建目标编号功能文件夹，更新其中的 BDD.md、ENGINEERING_SPEC.md 和 CHANGELOG.md。推导主成功场景、分支场景、异常场景和失败测试映射；完成文档后先停止，汇报文档结果和下一步实现计划，等待我确认。确认后展示预期 red failure，实现最小改动，运行验证并记录证据，最后把完成的功能分支合并到 dev。不要创建单独规划文档。
```

## 专项提示词

Python ML 工作组合 `$cx-bdd`、`$cx-tdd`、`$cx-pytorch-tdd`。

Rust 工作组合 `$cx-bdd`、`$cx-tdd`、`$cx-rust-tdd`。

最终审查使用 `$cx-evidence`，或明确启动只读 subagent `cx-review`。
