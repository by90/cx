# 用例驱动 TDD AI 辅助开发流程

## 核心原则

用例文档定义主成功场景和分支场景，任务文档定义一个类或一组类的实现边界，变更文档定义 AI 当前应该如何继续工作。TDD 用可执行测试证明任务行为。

## 标准顺序

1. 使用 `$cx-workflow` 判断任务类型并创建可见待办事项。
2. 开工前询问执行模式：直接做完文档、测试、实现和验证，或每完成一个任务后征求同意。
3. 如果用户没有明确选择逐任务确认，默认直接做完。
4. 使用 `$cx-story` 阅读 `docs/cx` 项目说明和相关主成功场景。
5. 若存在未完成变更，先阅读 `changes/`，并以未完成变更决定当前工作。
6. 如果需要新增用例、分支或任务，先更新 `00.用例.md`、`00. 设计.md` 或 `tasks/`。
7. 如果需要调整已有任务，写入 `changes/`，说明时间戳、任务编号、任务名称、之前做了什么、现在应该如何。
8. 文档完成后不作为默认停止点；继续使用 `$cx-tdd` 写最窄失败测试。
9. 实现一个任务涉及的一份代码文件，并在必要时更新一个一一对应的单元测试文件。
10. 运行最窄验证，记录命令和结果。
11. 只有逐任务确认模式才在任务完成后等待用户核对。

## 文档位置

```text
docs/cx/00.项目说明.md
docs/cx/01.创建用户/00.用例.md
docs/cx/01.创建用户/00. 设计.md
docs/cx/01.创建用户/tasks/01.编写用户实体/00.任务.md
docs/cx/01.创建用户/changes/20260629T120000-任务01-编写用户实体.md
```

## 推荐提示词

```text
请使用 $cx-workflow 和 $cx-story。开工前先询问执行模式；如果我没有选择逐任务确认，默认直接做完文档、测试、实现和验证。先检查 docs/cx 下目标场景的 changes 是否有未完成变更，再阅读 00.用例.md、00. 设计.md 和当前任务文档，然后使用 $cx-tdd 写失败测试并实现当前任务。
```

Python 或 PyTorch 工作组合 `$cx-story`、`$cx-tdd`、`$cx-pytorch-tdd`。

Rust 工作组合 `$cx-story`、`$cx-tdd`、`$cx-rust-tdd`。
