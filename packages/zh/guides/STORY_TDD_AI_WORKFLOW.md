# 用例驱动 AI 辅助开发流程

## 核心原则

用例文档定义主成功场景，以及挂靠到具体主步骤的条件、替代和异常子步骤；任务文档定义一个类或一组类的实现边界，变更文档定义 AI 当前应该如何继续工作。默认执行一个任务文档后只编辑一个生产代码文件；单元测试和 TDD 只在明确声明时进入任务范围。

## 标准顺序

1. 使用 `$cx-workflow` 判断任务类型并创建可见待办事项。
2. 默认执行模式是单任务单代码文件：完成当前任务文档后只编辑该任务绑定的一个生产代码文件。
3. 只有用户明确要求连续推进多个任务时，才继续处理下一个代码文件；只有明确要求逐任务确认时，才在每个任务后等待核对。
4. 使用 `$cx-story` 阅读 `docs/cx` 项目说明和相关主成功场景。
5. 若存在未完成变更，先阅读 `changes/`，并以未完成变更决定当前工作。
6. 如果需要新增用例、条件子步骤或任务，先更新 `00.用例.md`、`00. 设计.md` 或 `tasks/`。
7. 如果需要调整已有任务，写入 `changes/`，说明时间戳、任务编号、任务名称、之前做了什么、现在应该如何。
8. 任务文档完成后，实现该任务涉及的一份生产代码文件。
9. 默认不创建或修改单元测试；只有明确声明 TDD、单元测试或失败测试时，才使用 `$cx-tdd` 并更新一个一一对应的单元测试文件。
10. 运行最窄验证，记录命令和结果。
11. 使用 `$cx-evidence` 做代码完成后的强制 review；review 不通过时任务保持未完成并回到修复。
12. review 通过后默认汇报并停止在当前代码文件边界。

## 文档位置

```text
docs/cx/00.项目说明.md
docs/cx/01.创建用户/00.用例.md
docs/cx/01.创建用户/00. 设计.md
docs/cx/01.创建用户/tasks/01.编写用户实体/00.任务.md
docs/cx/01.创建用户/changes/20260629T120000-任务01-编写用户实体.md
```

## 推荐提示词

```text
请使用 $cx-workflow 和 $cx-story。先检查 docs/cx 下目标场景的 changes 是否有未完成变更，再阅读 00.用例.md、00. 设计.md 和当前任务文档。默认完成当前任务文档后只实现该任务绑定的一个生产代码文件；默认不创建或修改单元测试。只有我明确要求 TDD、单元测试或失败测试时，才使用 $cx-tdd。代码写完后必须用 $cx-evidence review；review 不通过时任务不得标记完成。
```

Python 或 PyTorch 默认工作组合 `$cx-story`；明确声明测试时叠加 `$cx-tdd`、`$cx-pytorch-tdd`。

Rust 默认工作组合 `$cx-story`、`$cx-rust-tdd`；明确声明测试时叠加 `$cx-tdd`。
