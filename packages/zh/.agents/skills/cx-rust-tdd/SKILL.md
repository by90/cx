# cx Rust 测试补充

## 边界

本技能不负责 Rust 通用实现流程，也不重新定义测试先行步骤。先完整执行 `$cx-tdd`，再把本技能的 Rust 规则叠加到同一个测试循环中。

没有明确测试要求时，不使用本技能，不创建、不修改、不运行 Rust 单元测试。

## Rust 测试工具

- 纯模块行为使用 Rust 内置 `#[cfg(test)]` 与 `#[test]`；跨模块公共入口使用 `tests/` 下的集成测试；需要验证公共示例可编译时使用文档测试。
- 失败阶段运行最窄的 `cargo test <filter>` 并记录目标失败；通过阶段先重复同一命令，再按影响范围运行 `cargo test`。
- 修改 Rust 代码后运行 `cargo fmt --check`，可行时运行 `cargo clippy --all-targets --all-features`。
- 测试成功值、错误枚举、边界值和所有权敏感行为；断言必须遵守 `$cx-tdd` 的常量右值和有序集合抽样规则。
- 不用 `unwrap`、`expect` 或 `panic!` 掩盖生产错误；需要验证失败时直接匹配明确的 `Result` 或错误枚举。

## 真实测试数据

- 数据相关 Rust 测试必须使用测试数据库中的真实数据，不在测试模块中临时制造替代数据。
- 数据库打开、固定范围读取和共享领域对象构造集中在一个公共测试夹具模块，例如 `tests/common/mod.rs` 或项目已有的等价入口。
- 公共夹具使用 `std::sync::OnceLock` 或项目已有的一次初始化机制，保证同一测试进程只读取一次；各测试模块只复用共享对象，不得各自打开数据库或重复构造同一数据。
- 测试数据库或所需记录缺失时报告前置条件缺失，不用内存数据库、伪仓库或模拟访问层替代。
- 除非用户当前请求明确要求，禁止 mock、test double 和伪造仓库测试。

## 输出

除 `$cx-tdd` 要求的记录外，补充记录最窄 `cargo test` 命令、完整测试结果、共享测试数据库入口、`cargo fmt --check` 结果，以及 `clippy` 结果或未运行原因。
