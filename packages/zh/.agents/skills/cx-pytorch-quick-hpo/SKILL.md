# cx PyTorch 快速调参

## 目的

快速调参不是追求一次性得到最终最优模型，而是在可控成本下排除错误方向，找出值得进入全量调参的候选 recipe。优先确定数据 recipe：字段启用/禁用、字段分级数组、特征模式、标签定义和窗口长度；然后确定训练 recipe：batch size、learning rate、optimizer、scheduler；最后搜索模型容量和结构。

## 硬规则

1. 训练、推理和回测入口必须能作为独立脚本运行，并在默认配置下完成一次有效执行；菜单或调参器只能调用这些脚本或它们背后的稳定入口。
2. 调参脚本只允许通过 typed config hook 在内存中复制并修改配置，再把配置对象传给训练、推理或回测入口；不要直接改写当前配置文件作为 trial 行为。
3. Python 业务脚本、训练脚本、诊断脚本和一次性脚本不得用命令行参数改变行为；需要调整行为时，在 config 子系统定义带默认值的配置项。
4. 快速调参使用约十分之一完整实体样本，抽样单位必须是完整实体。例如股票时间序列要抽股票而不是抽零散日期，并记录实体数量、覆盖范围和随机种子。
5. Tensor 默认使用 `float32`；除非已说明理由，不要让数据管线或中间统计默默变成 `float64`。
6. 发现坏数据、越界分级、NaN、极端值或 shape 错误时，必须定位出错实体、日期、字段和值并解释原因；不要用过滤失败样本的方式掩盖问题。
7. 字段组合必须以“字段启用/禁用数组 + 显式字段顺序”表达。可以给候选 recipe 起可读名称，但持久化结果必须保存具体字段清单，不要只保存 `daily_fields` 这类模糊名称。
8. 字段分级必须以“每个字段选择几级 + 每级边界数组”表达。不同字段可以使用不同级数和不同边界；trial summary 必须保存最终展开后的 per-field bins。
9. 分级数组先由字段业务含义、理论最小/最大值、实际分布、中心区间宽度和极端值过滤规则确定；训练时优先搜索每个字段使用 7/9/11 还是一侧压缩后的级数，而不是临时乱改边界。
10. 有正负方向的相对指标通常使用 7/9/11 奇数级；只有单向非负或非正指标时，级数应压缩为一侧等级并保留“无/中性”含义。
11. 标签定义、标签类别选择和“最高/最低几级算机会或风险”的规则属于 series/数据层共享语义，不允许在单个模型层硬编码。
12. 特征模式要作为 recipe 维度研究：原始值、相对值、分级值、归一化值、排名值可以分别或分阶段比较；标签应保持同一口径，避免把特征模式变化误当成标签变化。
13. 排名更像横截面归一化阶段，不应默认永久混入所有基础字段。先验证基础字段、位置字段、排名变换、基础+排名是否各自有效，再决定是否保留排名字段。
14. 窗口长度（常见配置名为 `window_size` 或 `window_length`）和 batch size 必须绑定搜索。窗口越长，显存和计算量越高，尤其含 attention/pair 表征时可能接近随窗口平方增长；不要把短窗口的 4096/8192 batch 直接套到 20/40/60 长窗口。
15. 快速调参必须覆盖短、中、长窗口。除非项目配置或显存明确不允许，窗口候选至少包含当前基准附近长度、`20`、`40` 和 `60`；长窗口要用显存预算缩小 batch 后验证。
16. 在双 NVIDIA RTX 5090、单卡 32GB 显存这类高显存环境中，快速调参必须把显存利用率、GPU 占用率和训练吞吐作为一等约束。每个窗口/字段 recipe 都优先寻找最大稳定 batch：可行时从 `8192` 开始，不稳定或 OOM 时依次降到 `4096`、`2048`、`1024`、`512`；只有窗口过长、字段过多或模型结构导致爆显存时才逐级降低 batch，不要为了保守而长期使用小 batch 慢训。
17. batch size 和 learning rate 必须成对比较。batch 变大时可以略微增大学习率，batch 因长窗口变小时应重新验证学习率，不要孤立比较 batch 或 learning rate。
18. 分类任务默认以 `val_loss`/cross-entropy 作为 objective、调参和 early stopping 监控指标；`test_loss` 是泛化确认指标，回测是业务确认指标。
19. 当不同分级数量导致类别数不同，`val_loss` 的绝对值不一定可直接横比；同一标签/分级口径内先比 loss，跨口径时同时看 test_loss、改进次数、校准后的回测和错误分布。
20. 快速训练上限通常是 60 epochs，early stopping 监控 `val_loss`，patience 一般从 8 开始；自然跑满 60 epochs 且持续改善是候选质量信号，但不是唯一准入条件。
21. 多模型项目中，每个模型都可能有自己的最佳字段、分级、窗口和容量。保留模型内最佳候选，不要过早只保留一个跨模型全局排名。

## 推荐流程

1. 先读项目 `AGENTS.md`、`docs/INDEX.md`、目标功能组 `ENGINEERING_SPEC.md`、`CHANGELOG.md` 和已有实验摘要；重新开始调参时从当前仓库配置和已保存结果出发，不继承旧线程里的临时判断。
2. 检查是否有旧调参进程仍在运行；长时间 quick 必须按项目规则使用可见外部终端和临时防睡眠机制，并在结束、失败或交接前清理。
3. 用 `$cx-common-module` 搜索现有训练入口、推理入口、回测入口、config hook、recipe 持久化和指标汇总能力；已有公共入口优先复用。
4. 用 `$cx-pytorch-tdd` 为配置映射、per-field bins、feature mode、窗口/batch 绑定、输出持久化和指标解析补最窄测试；训练测试保持 CPU、小模型、小数据。
5. 做字段语义和相关性研究：记录每个字段的中文业务含义、理论最小/最大值、实际 min/max、分位数、中心区间、极端值规则、与标签的正负贡献和噪声嫌疑。
6. 生成字段候选时，用启用/禁用数组表达字段组合，至少分别验证：基础日线字段、基础日线+市场强度、基础日线+位置字段、基础日线+市场强度+位置字段、排名变换阶段、基础+排名组合。
7. 生成分级候选时，为每个字段准备 7/9/11 基准边界和必要的中心宽/窄变体；中心持平区间通常优先覆盖最宽分布区，两侧按业务值对称或近似对称。
8. 第一阶段固定模型参数和训练参数，只比较字段组合、字段分级数量/边界、特征模式和标签口径。该阶段不要同时搜索模型容量，避免把数据 recipe 和模型容量混淆。
9. 第二阶段固定数据 recipe，搜索窗口长度与 batch size。以当前最佳窗口为基准，同时覆盖 `20`、`40`、`60`；在双 RTX 5090 32GB 环境中，按 `8192`、`4096`、`2048`、`1024`、`512` 的顺序为每个窗口寻找最大稳定 batch，以不 OOM、显存利用率充分、GPU 占用率高和平均 epoch 秒数最低为准。长窗口、字段过多或模型结构导致 OOM 时才逐级下调，不要从低 batch 开始保守验证。
10. 第三阶段固定数据 recipe 和窗口/batch 预算，搜索 learning rate、weight decay、optimizer、scheduler、risk penalty 等训练超参；batch 增大时同步尝试略高 learning rate。
11. 第四阶段才搜索模型结构和容量，例如 hidden size、层数、heads、dropout、模板数、pattern 数、diffusion steps、encoder/decoder 深度。
12. 每个阶段至少复跑当前 best recipe 一次，确认不是偶然随机结果；进入下一阶段前，把 top candidates、失败 trial、pruned trial 和排除理由写入目标文档集或实验摘要。
13. 快速调参完成后，选择 3 个候选进入全量调参：优先 `val_loss` 最低且 `test_loss` 接近；loss 接近时优先改进次数更多、未早停且回测更稳的候选。

## 评价口径

- `val_loss` 第一优先：决定训练目标和 early stopping，不应被单次高回测覆盖。
- `test_loss` 第二优先：用于判断 quick 的 val 改善是否能泛化。
- 回测第三优先：用于业务验证；当 `val_loss`/`test_loss` 相近时，回测结果可以决定排序。
- 改进次数和跑满情况是收敛质量信号：60 epochs 中持续改善通常优于很早停止；但如果 loss 明显更差，不能因为跑满而晋级。
- 回测很高但 loss 明显更差时，优先怀疑偶然性、标签错位、分级边界、样本选择或收益统计口径。

## 必须保存的证据

- 训练、推理和回测脚本在默认配置下可独立运行的命令和结果。
- quick 抽样实体、样本比例、时间覆盖、数据版本、随机种子和硬件信息。
- 每个字段的业务含义、理论范围、实际分布、极端值处理、相关性和是否启用。
- 每个 trial 的字段启用数组、字段顺序、feature mode、per-field level count、展开后的 per-field bins、标签口径。
- 每个 trial 的窗口长度、batch size、峰值显存、显存余量、GPU 占用率、是否 OOM、样本吞吐和平均 epoch 秒数。
- 每个 trial 的 optimizer、learning rate、weight decay、scheduler、模型结构参数和随机种子。
- `best_val_loss`、`final_val_loss`、`test_loss`、回测结果、训练 epochs、早停 epoch、改进次数、改进比例。
- top 3 候选、复跑结果、进入全量调参理由，以及未进入全量调参候选的排除理由。
