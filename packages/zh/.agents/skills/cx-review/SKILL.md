# cx 交付物 review

## 目的

在任何交付物完成后、任务或变更标记完成前执行本地 review。交付物包括代码、用例文档、任务文档、变更文档、设计文档、教程、研究报告、发布说明和流程规则。任何 P0/P1/P2 发现都表示 review 不通过，当前任务、变更或交付物必须保持未完成，修复后重新 review。

## 通用规则

1. 先确定本轮实际交付物类型；一次触及多个类型时，逐类 review。
2. review 必须对照用户请求、`docs/cx` 项目说明、用例、设计、任务、变更和相关来源，不能只看输出表面。
3. review 不通过时，输出具体发现、证据和修复方向，不得把任务、变更或交付物标记完成。
4. review 通过后，把 PASS 结论、审查范围、验证证据和剩余风险写入当前任务或变更文档；没有目标 cx 文档时，在最终摘要中说明。

## Review 类型

### 代码

- 实现是否与用例、设计、任务和变更文档完全一致，没有少做、超做或改变业务含义。
- 是否存在重复判断、重复转换、重复配置读取、重复字段传递、相似 helper 或多个局部变量表达同一概念。
- 是否完整 OOP；涉及状态、生命周期、不变量和领域协作时，必须由明确对象或等价类型表达。
- 是否极简；没有多余校验、多余兜底、多余异常包装、多余变量传递、多余参数、多余重复命名、臃肿文件或无复用价值抽象。
- 构造函数和普通函数是否用默认参数表达配置默认值，例如 `path=Config.default_config_file()` 或 `batch_size=config.train.batch_size`；函数体内是否用同名字段承载参数，例如 `self.batch_size = batch_size`。

### 文档

- 文档是否有明确读者、目标、范围、状态和唯一存放位置。
- 内容是否与项目说明、用例、设计、任务、变更、代码行为和验证证据一致。
- 是否没有散落文档、重复说明、过期说法、互相冲突的规则、含糊待办或没有依据的结论。
- 结构是否短小清楚，能指导下一步行动，而不是堆叙述或重复背景。
- 是否只写具体事实、具体动作和具体决策；是否没有空泛铺垫、套话、重复目标、遗漏“要做什么”或自造未定义术语。
- 项目说明、用例、设计和任务是否各自承担自己的内容；是否没有把同一套目标重复写到多处。
- 用例是否只表达业务场景、主成功场景、条件子步骤和子用例；是否没有把测试计划、实现任务或设计说明写进用例正文。
- 任务文件是否使用 `tasks/NN.中文任务名.md`，变更文件是否使用 `changes/中文变更名.md` 且不带时间戳；变更文档是否只记录已经实施后的后续变化。
- `src/<subsystem>/` 下的通用包是否有包内 `readme.md`；readme 是否只列公共接口和使用方式，没有把实例配置节、内部字段或实现步骤逐项写成接口文档。

### 教程

- 教程是否能按顺序执行；前置条件、命令、输入、预期输出和失败处理是否明确。
- 命令、路径、界面文案和 API 是否与当前项目真实状态一致。
- 是否没有隐藏步骤、跳步、过时命令、营销式介绍或把一次性经验伪装成稳定流程。
- 示例是否最小可运行，且不会引导用户破坏工作区、泄露凭据或绕过项目规则。

### 研究

- 是否先定义研究问题、时间窗口、纳入标准、排除标准和目标读者。
- 每个非显然主张是否有可靠来源；最新、推荐、价格、规则、模型能力或论文状态等易变信息是否已联网核对。
- 是否区分一手来源、论文、博客解读、厂商声明和社区弱信号。
- 是否给出综合判断、限制、未知项和可执行建议，而不是只罗列材料。

### 设计

- 设计是否从目标行为、业务约束、数据边界、生命周期和不变量出发。
- 是否说明公共入口、常规调用方式、特殊场景入口、状态来源、复用边界、非目标和验证方式。
- 是否比较必要取舍，并记录为什么不采用更复杂、更动态或更兼容旧路径的方案。
- 是否可由后续一个任务、一个代码文件或明确拆分后的任务序列落地。

### 流程与变更

- 变更文档是否准确说明之前状态、当前要求、顺序任务清单、影响文件、验证证据和完成状态。
- 任务或变更是否只有在对应交付物 review PASS 后才标记完成。
- 是否没有把失败、缺证据、未验证、未审查或用户未确认的内容写成已完成。

## 输出格式

```text

   Evidence:
   Fix:

- Documents checked:
- Commands or sources checked:

Review decision:
- PASS 或 FAIL
- FAIL 时任务、变更或交付物必须保持未完成

Residual risk:
- ...
```
