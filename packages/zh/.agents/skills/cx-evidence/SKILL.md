# cx 证据审查

## 目的

审查工作是否真的由 `docs/cx` 下的 story、任务和变更文档驱动，并且是否遵守默认单任务单代码文件、显式测试、完整 OOP、极简复用、业务语义和验证证据纪律。这是代码完成后的强制 review 门槛，不是只看风格的 review。任何 P0/P1/P2 发现都表示 review 不通过，当前任务不得标记完成；必须修复后重新 review。

## Review 重点

1. 文档一致性：实现是否与 `00.用例.md`、`00. 设计.md`、当前任务文档和当前变更文档完全一致；不能少实现、超范围实现或改变业务含义。
2. 业务语义：类、方法、状态、错误暴露和数据流是否符合领域语言；不能只是测试能过或技术上能跑。
3. 重复味道：是否出现相同判断、相同转换、相同配置读取、相同字段传递、相似 helper 或多个局部变量表达同一概念；有味道时先用 `$cx-common-module` 判断复用入口。
4. 完整 OOP：涉及状态、生命周期、不变量、领域协作时，是否由明确类、dataclass、protocol、struct、enum、trait 或方法表达；不能用松散字典、魔法字符串、字符串分发或动态属性访问替代。
5. 极简实现：是否没有多余校验、多余兜底、多余异常包装、多余变量传递、多余参数、多余重复命名、过长文件、过长变量名、句子式标识符和没有真实复用价值的抽象。

## 审查清单

1. 所有 cx 场景、任务和流程文档是否都在 `docs/cx/` 下？
2. `docs/cx/` 之外是否没有被当作 cx 流程文档维护的内容？
3. 是否存在 `docs/cx/00.项目说明.md`？
4. 每个主成功场景是否有 `00.用例.md`、`00. 设计.md`、`tasks/` 和 `changes/`？
5. 当前工作是否先检查并处理了 `changes/` 中未完成变更？
6. 如果一个变更涉及多个任务或用例文档修改，是否存在顺序任务清单？
7. 是否记录了本轮执行模式；默认模式下是否只完成当前任务文档并编辑一个生产代码文件；如连续推进多个任务，是否有用户明确要求？
8. 每个任务是否有 `tasks/<编号.任务名>/00.任务.md`？
9. 每个任务是否声明了类或类型组作为任务量具？
10. 每个任务是否只绑定一个生产代码文件；若超出范围，是否拆成多个任务或写明理由？
11. 条件、替代和异常是否用 `1.1`、`2.1` 这类子编号写在 `00.用例.md` 的对应主成功步骤下，而不是创建分支文件夹或堆叠独立分支章节？
12. 是否没有创建草稿、备份、旧版、并行计划或重复说明文档？
13. 单元测试或 TDD 是否由用户请求、任务文档或变更文档明确声明；未声明时是否没有创建或修改单元测试？
14. 明确要求测试时，是否展示了预期 red failure 并记录测试命令和结果；未要求测试时，是否记录了其它验证方式？
15. 长时间运行、构建、测试、安装或 UI 实机检查是否启动了项目提供的临时防睡眠机制，并在收尾前关闭？
16. UI 相关修改是否按项目方式打包、安装或启动真实应用，并在 macOS GUI 项目中使用 Computer Use 或项目指定方式完成实机观察？
17. 可复用功能、类或组件是否先搜索已有实现、相关 skills 和当前 `00. 设计.md` 中的复用记录，再抽取或说明不抽取理由？
18. 明确要求 Python 单元测试时，是否使用 `unittest`，除非项目有例外？
19. Python 源码是否位于 `src/<subsystem>/`；明确要求测试时，测试是否在 `tests/` 下镜像 `src` 结构，并按 `xx.py` -> `xx_test.py` 一一对应？
20. 新增或修改的源码文件和明确声明的单元测试是否都有文件级作用说明，并说明本文件维护的主要类、函数或测试对象？
21. 新增或修改的类、函数、方法和明确声明的测试方法是否都有职责说明、参数含义说明、返回值说明或明确无返回值说明？
22. 新增或修改的源码和明确声明的单元测试是否都遵守逐行意图注释标准，且每一行业务逻辑都有相邻说明注释？
23. Python 修改后是否运行了 Black 默认规范检查，例如 `python -m black --check src tests tools` 或项目等价命令？
24. Python 默认行为是否通过默认参数、配置对象、dataclass、factory 或小方法表达，而不是在 `__init__` 中堆叠大量分支？
25. Python 脚本是否没有新增命令行参数解析，且所有可调整行为都通过 config 子系统配置项表达，并且每个配置项都有默认值？
26. 实现是否逐条覆盖当前任务文档的预期行为，并且没有实现任务文档之外的功能？
27. 实现是否与 `00.用例.md` 的主成功场景和条件子步骤一致，尤其是成功路径、异常暴露方式和结束条件？
28. 实现是否与 `00. 设计.md` 的复用入口、公共代码使用、设计决策和非目标一致？
29. 业务命名和对象职责是否表达领域含义，而不是只表达技术步骤或测试细节？
30. 是否遵守对应实现 skill 的最小实现纪律：绝对禁止臃肿代码，未提前框架化、泛化、抽象或增加需求外验证？
31. 是否没有捕获、吞掉、静默跳过、默认值兜底、空结果兜底、假装重试成功或包装任何会造成产品问题的异常；如存在处理，是否有明确业务要求、用户可见路径和验证？
32. 是否没有在大循环、训练循环、热路径或批处理内逐条检查数据合法性来兜底、过滤或拖慢性能？
33. 是否不存在多余校验、多余预检、多余中间变量、多余参数传递、多余重复变量名或参数名？
34. 是否不存在重复代码味道；如果存在相似逻辑、相似变量组、相似配置读取或相似转换，是否已经用 `$cx-common-module` 审视公共入口？
35. 代码是否保持极简、短小、直接，并且可复用功能、类或逻辑已经先使用 `$cx-common-module` 搜索和登记？
36. 是否采用完整 OOP：Python 用类、dataclass、protocol 或显式对象协作表达状态和不变量；Rust 用 struct、enum、trait 和方法表达领域对象？
37. 是否避免了非 OOP 的动态访问方式，例如 `getattr`、`setattr`、`delattr`、monkey patch、动态注入或字符串分发；如不可避免，是否记录理由、隔离实现并验证？
38. 通用功能、可复用功能或可复用类是否先明确公共入口、常规调用方式、特殊场景入口、实例或状态生命周期、状态来源、验证如何覆盖所有源码调用和非目标？
39. 明确要求测试时，测试是否覆盖特殊入口、生命周期、状态恢复或测试隔离方式？
40. 是否存在只有一个调用点且没有隔离真实复杂度的 helper、包装函数、校验器或常量？
41. 是否存在需求未要求的验证、异常包装、动态构造、扫描器、注册表或未来扩展入口？
42. Rust 代码变更是否运行了必要 cargo 命令？
43. 文件、类、方法和变量命名是否短而清楚，避免过长描述性标识符；文件是否保持高内聚、不过度膨胀？
44. review 是否通过；如不通过，当前任务和变更是否仍保持未完成状态？
45. 工作是否满足提示词契约：目标、上下文、约束、必须遵循的流程、验证方式和交付物？
46. 最终摘要是否报告已运行命令、结果、review 结论、跳过检查及原因、残留风险？
