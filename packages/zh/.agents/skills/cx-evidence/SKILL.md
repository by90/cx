# cx 证据审查

## 目的

审查工作是否真的有测试和目标研发文档集支撑。这是交付门槛，不是只看风格的 review。

## 审查清单

1. 每个变更是否都有 `CHANGE-*` 条目？
2. 每个 `CHANGE-*` 条目是否映射到同一文档集的 `ENGINEERING_SPEC.md`？
3. 行为变化是否新增或更新了 BDD 场景？
4. 每个 BDD 场景是否映射到测试？
5. 文档完成后、测试或实现开始前，是否有用户明确确认？
6. 工作开始时是否在对话中创建了待办事项列表，并在过程中逐项更新到完成、取消或明确阻塞？
7. 实现前是否展示了预期 red failure？
8. 测试命令和结果是否记录？
9. 长时间运行、构建、测试、安装或 UI 实机检查是否启动了项目提供的临时防睡眠机制，并在收尾前关闭？
10. UI 相关修改是否按项目方式打包、安装或启动真实应用，并在 macOS GUI 项目中使用 Computer Use 或项目指定方式完成实机观察？
11. 是否把 `docs/` 根目录限制为索引、说明和版本索引，并把具体文档放入 `docs/001_feature_name/` 这类带三位序号、小写、下划线分隔的功能组目录？
12. 是否创建了孤立规划文档？
13. 可复用功能、类或组件是否先搜索已有实现、相关 skills 和 Reusable Capability Registry，再抽取或说明不抽取理由？
14. Python 测试是否默认使用 `unittest`，除非项目有例外？
15. Python 源码是否位于 `src/<subsystem>/`，测试是否在 `tests/` 下镜像 `src` 结构，并按 `xx.py` -> `xx_test.py` 一一对应？
16. 新增或修改的源码文件和单元测试是否都有文件级作用说明，并说明本文件维护的主要类、函数或测试对象？
17. 新增或修改的类、函数、方法和测试方法是否都有职责说明、参数含义说明、返回值说明或明确无返回值说明？
18. 新增或修改的源码和单元测试是否都遵守逐行意图注释标准，且每一行业务逻辑都有相邻说明注释？
19. Python 修改后是否运行了 Black 默认规范检查，例如 `python -m black --check src tests tools` 或项目等价命令？
20. Python 默认行为是否通过默认参数、配置对象、dataclass、factory 或小方法表达，而不是在 `__init__` 中堆叠大量分支？
21. Python 脚本是否没有新增命令行参数解析，且所有可调整行为都通过 config 子系统配置项表达，并且每个配置项都有默认值？
22. 是否遵守对应实现 skill 的 `## 最小实现纪律`：绝对禁止屎山代码，未提前框架化、泛化、抽象或增加需求外验证？
23. 代码是否保持极简、短小、直接，并且可复用功能、类或逻辑已经先使用 `$cx-common-module` 搜索和登记？
24. 是否避免了非 OOP 的动态访问方式，例如 `getattr`、`setattr`、`delattr`、monkey patch、动态注入或字符串分发；如不可避免，是否记录理由、隔离实现并测试？
25. 通用功能、可复用功能或可复用类是否先明确公共入口、常规调用方式、特殊场景入口、实例或状态生命周期、状态来源、测试如何覆盖所有源码调用和非目标？
26. GUIDE 或 README 是否给出常规调用示例，并且测试覆盖了这个示例？
27. 测试是否覆盖特殊入口、生命周期、状态恢复或测试隔离方式？
28. 是否存在只有一个调用点且没有隔离真实复杂度的 helper、包装函数、校验器或常量？
29. 是否存在需求未要求的验证、异常包装、动态构造、扫描器、注册表或未来扩展入口？
30. 新增同类能力、配置节、字段或数据源时，是否避免修改硬编码导出逻辑或流程分支？
31. Rust 代码变更是否运行了必要 cargo 命令？
32. 工作是否满足提示词契约：目标、上下文、约束、必须遵循的流程、验证方式和交付物？
33. 普通、非编程任务是否没有自行创建 BDD；若边界不清，是否先询问用户？
34. 最终摘要是否报告已运行命令、结果、跳过检查及原因、残留风险？

## 输出格式

```text
Findings:
1. [severity] file:line - issue
   Evidence:
   Fix:

Verified:
- command -> result

Missing evidence:
- ...
```
