# cx PyTorch 全量调参

## 目的

把快速调参得到的候选配置放到完整数据上验证，选出最优到第 3 名方案，并在训练、测试、回测和用户确认后把最佳模型作为 release 候选保存。

## 铁律

1. 全量调参必须以 `$cx-pytorch-quick-hpo` 的 best recipe 和候选列表为输入；如果跳过快速调参，必须在目标文档集记录理由、风险和替代验证方式。
2. 训练脚本必须能单独运行，并在默认配置下完成一次训练；全量调参脚本只能通过 typed config hook 在内存中复制并修改配置，再把该配置对象传给训练入口。
3. 回测脚本必须能单独运行，并在默认配置下完成一次回测；全量调参脚本只能通过内存配置对象调用回测入口，不能直接修改当前回测配置文件。
4. 调参脚本保存结果时，只能把 trial 配置、full-data recipe、测试指标、回测配置和模型产物路径持久化到输出目录；禁止直接改写项目当前配置、默认配置或用户手动维护的配置文件。
5. 全量调参使用完整数据，计划上限是 600 个 epochs，并启用 early stopping 监控 `val_loss`，patience 一般设为 20；触发早停时必须停止，不得强行跑满 600。
6. 全量调参首先以快速调参最佳 `val_loss` 为参考标准，并辅以 full-data `val_loss`、`test_loss` 和回测结果评估候选。
7. 全量调参目标是找出最优到第 3 名方案；但必须先完整训练排名第 1 的方案，训练流程必须包含测试集评估。
8. 排名第 1 的方案完成训练和测试后，必须立即运行回测；如果回测结果好，再询问用户是否需要训练第 2 名和第 3 名候选进行比较。
9. 如果第 1 名方案回测结果不好，必须记录失败原因，并回到候选排序、字段/标签 recipe 或快速调参证据，而不是直接训练其他候选。
10. 最佳模型作为 release 候选保存时，必须遵守现有 `$cx-version` 流程、用户确认门禁、版本记录和发布证据要求。
11. 多模型项目必须先在每个模型内部选出模型内最优 release 候选，再用这些模型内最优方案比较出全局最优；全局最优默认用于推理。
12. 推理必须保留选择不同模型 release 的权利；根目录 `main` 终端菜单和每个模型 `main` 终端菜单都必须提供默认全局最优和可选历史/候选 release 的选择入口。

## 必须执行的流程

1. 读取快速调参的 best recipe、top candidates、字段建议、窗口长度、标签定义、训练超参和收敛证据。
2. 用 `$cx-common-module` 检查 full-data recipe、模型保存、测试评估和回测调用是否已有公共入口；没有时先定义调用模型。
3. 用 `$cx-pytorch-tdd` 验证完整数据配置构造、内存 config override、训练入口、测试阶段入口、回测入口和输出持久化。
4. 用完整数据训练第 1 名候选，并在训练流程中执行测试集评估，记录 `val_loss`、`test_loss` 和业务指标。
5. 训练完成后用同一份内存配置派生回测配置，运行回测并把回测配置和结果保存到输出目录。
6. 如果第 1 名候选回测结果达到目标，停止并询问用户是否训练第 2 名和第 3 名候选；用户确认前不要继续消耗算力。
7. 如果用户确认比较，再依次训练和回测第 2 名、第 3 名候选，并按 `val_loss`、`test_loss`、回测结果和业务风险综合排序。
8. 对多模型项目，先为每个模型选出模型内最优候选，再用模型内最优候选比较出全局最优；把全局最优设为默认推理 release。
9. 更新或要求更新根目录 `main` 终端菜单和每个模型 `main` 终端菜单：默认选中全局最优 release，同时允许用户选择其他模型 release 做推理。
10. 选定最佳模型后，把模型产物、recipe、测试结果、回测结果、数据版本、随机种子、全局最优依据、推理默认 release 和 release 建议写入目标文档集验证证据。

## 验证证据

- 快速调参 best recipe 和 top 3 候选来源。
- 训练脚本和回测脚本可以在默认配置下独立运行。
- 全量调参脚本只在内存中修改配置，输出目录包含 full-data recipe、测试配置、回测配置和候选结果，当前配置文件未被改写。
- 完整数据边界、训练/验证/测试切分、数据版本和随机种子。
- 600 epochs / patience=20 的执行情况、早停 epoch、best `val_loss` 和 `test_loss`。
- 第 1 名候选的训练、测试和回测结果，以及是否询问用户继续训练第 2/第 3 名候选。
- 第 2 名和第 3 名候选的训练比较结果；如果未训练，记录用户决定或停止理由。
- 每个模型的模型内最优 release 候选、全局最优选择依据和默认推理 release。
- 根目录 `main` 终端菜单和每个模型 `main` 终端菜单中，可选择默认全局最优或其他 release 的验证证据。
- 最佳模型的 release 候选路径、recipe、指标、回测结果和 `$cx-version` 后续动作。
