# cx 流程处理

## 目的

使用本 skill 作为 cx 工作流入口和 skills 命令官。它必须先判断用户请求属于项目说明、主成功场景、任务、变更、实现、验证、研究或发布中的哪一类，再推荐最小合适的 cx skills 组合和执行顺序。

## 核心规则

1. 项目所有 cx 场景、任务和流程文档都只放在 `docs/cx/` 下。
2. `docs/cx/` 之外的任何文档都与 cx 流程无关；除非用户明确要求处理非 cx 文档，否则不要读取、创建或维护它们来完成 cx 工作。
3. `docs/cx/00.项目说明.md` 是项目级说明入口；重要业务领域、术语和跨场景规则默认写入这里。
4. cx 文档必须写具体事实、具体动作和具体决策；一句话只表达一个意思，禁止空泛铺垫、重复目标、遗漏“要做什么”和自造未定义术语。
5. 项目说明只写项目目标、关键术语、设计约束和用例索引；目标用 `1. 2. 3.` 描述，每项目标必须说明要达到什么结果以及为什么需要该结果。
6. 用例只写参与者、前置条件、触发条件、主成功场景、挂到具体主步骤的条件子步骤、子用例和完成结果；不要在用例正文里写测试计划、实现任务或重复项目目标。
7. `docs/cx/01.创建用户/` 这类顶层编号文件夹表示主成功场景。
8. 每个主成功场景文件夹维护唯一当前 `00.用例.md`、唯一当前 `00.设计.md`、`tasks/` 和 `changes/`。
9. 条件、替代和异常步骤用 `1.1`、`2.1` 这类子编号写在 `00.用例.md` 的主成功场景对应步骤下，不创建分支文件夹。
10. 任务详情放在 `tasks/NN.中文任务名.md`，不要创建 `00.任务.md` 这类泛名文件。
11. 变更详情放在 `changes/中文变更名.md`，文件名不加时间戳；变更文档只记录已经实施后的后续变化，未实施、未提交或初始规划阶段不创建变更文档。
12. AI 决定当前做什么时，先看当前主成功场景 `changes/` 中未完成变更；只有没有未完成变更，或用户明确要求新增任务、新增条件子步骤或新增用例时，才规划新内容。
13. 任一时刻都只保留一份当前文档；不要创建草稿、备份、旧版、并行计划或重复说明文档。
14. 开发默认完整 OOP；Python 使用类、dataclass、protocol 或显式对象协作表达状态和不变量，Rust 使用 struct、enum、trait 和方法表达领域对象和边界。
15. Python 构造函数和普通函数优先用默认参数表达配置默认值，例如 `path=Config.default_config_file()` 或 `batch_size=config.train.batch_size`；函数体内用同名字段承载参数，例如 `self.batch_size = batch_size`。
16. `src/<subsystem>/` 下的通用包必须提供包内 `readme.md`，说明公共接口和使用方式；不要把实例配置节、内部字段或实现步骤逐项写成接口文档。
17. 代码默认极简、复用优先、低重复；避免臃肿文件、长串变量名、过度描述性标识符和没有真实复用价值的抽象。
18. 任务文档的基本量具是类或类型组；一个任务可以覆盖一个类，也可以覆盖一组高度内聚的类型组。
19. 默认执行节奏是完成一个任务文档后，只编辑该任务绑定的一个生产代码文件；如果需要第二个代码文件，先拆出下一个任务。
20. 默认不创建、不修改、不运行单元测试；只有用户当前请求、既有任务文档或变更文档明确声明需要单元测试或 TDD 时，才绑定一个对应测试文件并使用测试专项 skill。
21. 任何代码、文档、教程、研究、设计、流程变更或发布说明交付物完成后，必须先使用 `$cx-review` 做对应类型的本地 review，再认定任务、变更或交付物完成。
22. review 不通过时，当前任务、变更或交付物状态仍是未完成；必须先修复问题再重新 review。
23. 代码 review 重点核对实现是否与 `00.用例.md`、`00.设计.md`、当前任务文档和变更文档完全一致，是否存在重复代码味道，是否完整 OOP，是否极简且没有多余校验、多余变量传递、多余重复变量或参数命名，以及业务含义是否吻合。
24. 文档、教程、研究和设计 review 分别核对单一事实来源、教程可执行性、来源质量与综合结论、设计可落地性和业务含义；任一必需交付物 review 未通过时不得进入完成状态。

## 流程入口

1. 先判断任务类型：项目说明、主成功场景、条件子步骤、任务、变更、实现、缺陷修复、重构、研究、证据审查、发布版本或安装使用问题。
2. 对任何需要多步执行的任务，先在对话中创建待办事项列表；后续执行必须逐项更新状态，直到每项完成、取消或明确阻塞。
3. 对 cx 流程任务，先检查或创建 `docs/cx/` 和必要主成功场景文件夹。
4. 进入某个主成功场景后，先检查 `changes/` 是否有未完成变更。
5. 如果存在未完成变更，优先执行最早的未完成变更，不要新增任务或改写用例结构。
6. 如果没有未完成变更，再检查 `tasks/` 中未完成任务。
7. 只有用户明确要求新增任务、新增条件子步骤或新增用例时，才创建新的任务文档或修改用例结构。
8. 判断是否会改变代码行为、公共 API、数据结构、用户工作流、发布方式或研究结论。
9. 根据任务类型、影响范围、用户明确点名的 skill 和项目规则，推荐最小必要的 cx skills 组合和执行顺序；不要一次性套用所有 skills。
10. 给出当前步骤或直接执行；只有需求缺口会导致错误实现时才先提问。

## 强约束

1. 中文包是主源；修改 cx 规则时先完成中文技能、中文 AGENTS、中文模板和中文工具，再同步英文包。
2. 当前项目安装中文 cx 包时，所有 cx 生成或维护的文档必须使用简体中文；代码标识符、命令、API 名称和外部英文专名可以保留原文。
3. 使用中文写说明、计划、总结、文档或注释时，优先使用完整中文表达。
4. 用户要求提交、交付、发 PR 或发布时，把当前工作区视为一次整体变更：暂存已跟踪和未跟踪文件，创建一个提交；只有明显密钥、构建产物或无关大文件才需要先停下来确认。
5. 工作完成并经过用户确认后，相关工作分支应合并到 `main` 并删除本地分支；除非用户明确覆盖，否则不要把工作分支推送到远端。
6. 使用“验证证据”描述已经被文档、命令输出、人工检查、截图、测试或来源支撑的信息；单元测试只在明确声明时作为证据来源。
7. 任何实现、修复、重构、通用功能、可复用功能、可复用类或可复用能力抽取任务都必须遵守对应实现 skill 中的最小实现纪律。
8. 编程任务默认开发期快速失败：除非业务明确需要降级、恢复或用户可见提示，否则不得捕获、吞掉、兜底、静默跳过或包装会造成产品问题的异常。
9. 编程任务不得用大量预先检查、大循环内逐条合法性检查、热路径兜底或批处理静默过滤来掩盖数据错误或拖慢性能。
10. 通用功能、可复用功能或可复用类实现前，必须先确定调用模型，并写入 `00.设计.md`；只有已经实施后的后续变化才写入 `changes/` 变更文档。
11. 交付物完成后的 review 是任务、变更和交付物完成门槛，不是可选总结；review 发现任何文档不一致、重复味道、非 OOP、臃肿实现、多余校验、多余变量传递、多余命名、业务语义偏差、教程不可执行、研究来源不足或设计不可落地时，必须回到对应步骤修复。
12. 长时间运行、构建、测试、安装或 UI 实机检查前，必须查找并启动项目提供的当前平台防睡眠或会话保持机制；结束、阻塞或交接前关闭。
13. 运行 Python 脚本、测试、构建或工具命令时，必须使用 `uv` 安装或同步的 Python。
14. 在 Windows 原生 Codex App 中，默认使用 PowerShell 运行命令和本地设置脚本。

## 通用/可复用功能门禁

遇到通用功能、可复用功能、可复用类或共享工具时，先在当前主成功场景的 `00.设计.md` 中记录以下契约：

```text
公共入口：
常规调用方式：
特殊场景入口：
实例或状态生命周期：
状态来源：
验证如何覆盖所有源码调用：
readme 公共接口说明：
非目标：
```

- 公共入口说明调用方应该 import、构造或调用什么。
- 常规调用方式说明业务代码最常见、最短的用法。
- 特殊场景入口说明只有测试、不同数据源、不同配置目录或临时上下文才允许使用的入口。
- 实例或状态生命周期说明何时创建、缓存、重置、恢复和释放。
- 状态来源说明配置、文件、环境、数据库、设备探测或用户输入来自哪里。
- 验证覆盖方式说明如何让源码中的常规调用也被验证；只有明确要求单元测试时才写测试覆盖方式。
- readme 公共接口说明列出包内 `readme.md` 要公开的构造函数、方法、保护字段、全局切换和保存版本等入口。
- 非目标明确哪些验证、兼容层、动态扫描、注册表、未来扩展或异常包装本次不做。

## 提示词输入契约

当用户请求含混时，先把它整理成这个契约再规划：

```text
目标：
上下文：
分支：
约束：
必须遵循的流程：
验证方式：
交付物：
```

- 目标必须描述可观察行为，或明确的研究/发布结果。
- 上下文应标明 `docs/cx/` 主成功场景文件夹、任务文档、变更文档、目标代码文件、可选测试文件、分支和环境。
- 分支应标明本地工作分支名、合并目标 `main`，并说明远端只保留 `main` 和版本 tag。
- 约束应包含公开 API、语言规则、兼容性、性能、安全、完整 OOP、极简复用、单代码文件边界、是否明确要求单元测试或 TDD，以及任务量具。
- 必须遵循的流程要点名最小必要 cx skills；不要因为是代码实现就默认加入测试专项 skill。
- 验证方式列出期望命令、检查、截图或证据。
- 交付物说明要修改什么，以及最终摘要和证据要求。

只有缺失字段会造成错误行为、不安全改动或无法验证的工作时，才问澄清问题。否则直接执行，并简短说明假设。

## Skill 选择

- 用例发现、story、主成功场景、条件子步骤、任务文档或变更文档：使用 `$cx-story`。
- 用户或任务明确要求 TDD、单元测试、red-green-refactor 或失败测试：使用 `$cx-tdd`。
- 行为变更、缺陷修复、架构调整、实现规划：先用 `$cx-story` 处理 `changes/`，再按任务文档只实现一个代码文件；只有明确要求测试时叠加 `$cx-tdd`。
- 只更新变更文档或检查未完成变更：使用 `$cx-changelog`。
- 发布版本判断、SemVer 升级、tag 或 release notes：使用 `$cx-version`。
- 模型选择、模型原理、近期 AI 论文、学术/博客综合研究或研究报告：使用 `$cx-research`。
- 用户或任务明确要求 Python、PyTorch、Lightning、tensor 或 ML 单元测试时：叠加 `$cx-pytorch-tdd`。
- PyTorch 快速调参、字段贡献研究、特征组合、窗口长度、标签、训练超参、optimizer/scheduler 或模型容量初筛：叠加 `$cx-pytorch-quick-hpo`，并按需叠加 `$cx-pytorch-tdd`、`$cx-timeseries-modeling`、`$cx-research` 和 `$cx-common-module`。
- PyTorch 全量调参、完整数据训练、测试集评估、回测、top 3 候选比较或 release 候选模型选择：叠加 `$cx-pytorch-full-hpo`，并按需叠加 `$cx-pytorch-tdd`、`$cx-evidence` 和 `$cx-version`。
- 异构多变量时间序列、预测 horizon、协变量、backtesting 或时间序列框架选型：叠加 `$cx-timeseries-modeling`。
- Rust 实现或 Rust 测试：叠加 `$cx-rust-tdd`。
- 抽取或设计通用功能、可复用功能、可复用类、共享工具、稳定 API、复用组件、迁移重复逻辑：叠加 `$cx-common-module`。
- 代码、文档、教程、研究、设计、流程变更或发布说明完成后的强制 review：使用 `$cx-review`。
- 交付前检查、测试证据、review 结论、文档一致性、业务语义核对和剩余风险审查：使用 `$cx-evidence`。

## 执行顺序

1. 对多步任务，先在对话中创建待办事项列表；之后每完成、取消或阻塞一项，都要更新状态。
2. 对普通编程研发任务，先创建或切换到独立本地工作分支，再修改项目文件。
3. 使用 `$cx-story` 检查 `docs/cx/`，定位主成功场景文件夹，读取 `00.用例.md`、`00.设计.md`、`tasks/` 和 `changes/`。
4. 如果 `changes/` 有未完成变更，先按变更文档执行。
5. 如果没有未完成变更，再选择当前未完成任务。
6. 完成当前任务文档后，只编辑该任务绑定的一个生产代码文件；默认不写单元测试，除非当前请求、任务文档或变更文档明确声明。
7. 如果明确要求单元测试或 TDD，使用 `$cx-tdd` 和必要专项 skill 写失败测试、记录 red failure、实现最小改动并验证。
8. 任一交付物完成后必须使用 `$cx-review` 执行对应类型本地 review；review 必须先于任务、变更或交付物状态改为完成。
9. review 发现问题时，不要把任务、变更或交付物标记为完成；先修复实现、文档、教程、研究、设计或流程记录，再重新执行 review。
10. review 通过后，每个任务完成时更新任务文档、变更文档、review 结论和验证证据；默认不自动推进到下一个代码文件，除非用户明确要求连续处理后续任务。
11. 对已有实现或已有文档的收尾任务，先检查 `docs/cx/`、任务文档和变更文档，再使用 `$cx-review` 审查实际交付物，并在交付前使用 `$cx-evidence` 核验证据。
12. 对只读分析或代码审查，先读相关文件并列出风险，不做代码改动，除非用户要求修复。
13. 对可能复用的功能、类、组件、数据结构、测试夹具或 UI 状态模型，先使用 `$cx-common-module` 搜索已有实现和 registry，再决定是否新增抽象。
14. 对发布工作，使用 `$cx-version`；只有 `main` 可用于版本提交、带注释发布 tag 和发布 tag push。
15. 收尾前确认对话中的待办事项全部完成、取消或明确阻塞，并报告防睡眠机制关闭、验证结果、各类交付物 review 结论和剩余风险。

## 停止条件

- 缺少关键业务规则、目标环境或验收标准，并且继续实现会造成明显返工。
- 用户要求的实现与现有 `AGENTS.md`、技术约束或公开 API 冲突。
- 需要联网核对高风险或易变信息，但当前环境无法验证。

遇到停止条件时，先说明阻塞点和需要用户确认的最少问题。
