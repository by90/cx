# 专题：开发代码失败策略

## 解决的问题

防止开发代码通过自行校验、异常包装、错误兜底或旧接口兼容掩盖不能发布的问题。

## 适用范围

- 所有新增或修改的生产代码、脚本、工具、示例和明确声明的测试。
- 所有接口、参数、配置、路径、调用方和异常传播边界。

## 当前结论

除非用户在当前请求中明确要求具体校验或异常处理，代码不得自行增加一系列校验后主动抛出异常，也不得捕获、转换、包装、吞掉、静默跳过或兜底异常。底层异常必须保持原始类型、原始信息和原始调用栈，自然抛出并中止程序。

开发代码只实现当前最新意图。旧接口、旧入口、别名、适配层、桥接层、兼容分支、旧参数、旧配置、旧路径、旧行为和相关痕迹必须全部删除。调用方同步使用当前入口。

## 功能入口或源码位置

- `packages/zh/AGENTS.md`
- `SKILLS/zh/cx-workflow/SKILL.md`
- `SKILLS/zh/cx-review/SKILL.md`

## 输入与输出

- 输入：用户当前明确要求和底层库自然产生的错误。
- 输出：只实现当前意图的直接代码，以及未被拦截的原始异常。

## 关键约束

- 正常业务状态可以显式建模，程序异常不能转成业务分支继续运行。
- 用户明确要求的校验或异常处理只在指定范围内实现，不得自行扩大。
- 不通过兼容代码维持旧调用方。

## 最短使用方式

1. 直接调用底层能力。
2. 让底层异常原样传播。
3. 同步迁移调用方到当前入口。
4. 删除所有旧接口和兼容痕迹。

## 验证方式

- 搜索捕获、包装、自定义异常、默认兜底和兼容分支。
- 核对用户是否明确要求任何保留项。
- 检查调用方全部使用当前入口。
