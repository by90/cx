# 用例：维护 cx 工作流

## 目标

维护者修改 cx 流程后，安装包只保留最新规则，已有任务身份稳定，当前变更由 Git 保存，项目拥有按真实情况填写的 `AGENTS.md`，人工智能代理在实现前先读取当前领域的通用包教程并搜索公共入口。

## 参与者

- 主要参与者：cx 维护者
- 支撑系统：Git、技能安装脚本、工作流验证工具

## 前置条件

- 维护者位于 cx 源仓库的 `main` 分支。
- 工作区没有未说明的修改。

## 触发条件

- 维护者提出 cx 工作流规则调整。

## 主成功场景

1. 系统读取项目根目录 `AGENTS.md`，并打开当前领域登记的全部通用包教程。
1.1 如果项目 `AGENTS.md` 或通用包教程不存在，则系统使用 `$cx-doc` 的项目规则模板和教程规范补齐后返回第 1 步。
2. 系统搜索教程已经登记的通用包、稳定接口和真实调用点。
3. 系统检查当前主成功场景 `changes/` 中的临时变更文件。
3.1 如果存在临时变更文件，则系统以该文件作为当前工作指令。
4. 维护者更新当前用例、设计、原任务文档和实现，使所有正式文档只表达最新状态。
4.1 如果代码出现错误，则系统为对应原任务登记临时变更，并按同一流程修复。
4.2 如果实现方式改变，则系统修改原任务文档，不创建、删除或重命名任务文件。
5. 系统执行统一交付审查，检查交付物质量和完成证据。
5.1 如果审查不通过，则系统保留临时变更文件并继续处理。
6. 审查通过后，系统删除已完成的临时变更文件并把删除提交到 Git。

## 完成标准

- 四份固定任务文档全部反映当前有效规则。
- 每个通用包都有调用者教程，项目 `AGENTS.md` 能按领域导航全部教程。
- 中文源、英文源、安装包、模板和说明保持一致。
- `$cx-review` 已同时通过交付物质量审查和完成证据门禁。
- `changes/` 中没有已完成的变更文件。
