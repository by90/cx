# 变更：统一当前状态与审查流程

## 状态

统一审查中

## 关联对象

- 用例：`00.用例.md`
- 设计：`00.设计.md`
- 任务：`tasks/01.统一当前状态文档.md` 至 `tasks/04.同步双语安装包.md`

## 当前事实

- `changes/` 被当作长期审计记录，正式文档可以保留处理前后差异。
- 已有 story 的变化可以继续产生新任务，代码错误没有统一的临时变更入口。
- 交付物质量和完成证据由两个技能分别检查。
- 通用能力发现没有要求先读取专题文档，研究结论没有统一的项目目录。
- 开发规则允许自行增加校验、异常包装、兜底和兼容入口。
- 单一来源校验器不识别专题文档目录和当前临时变更结构。

## 目标状态

- 正式文档只表达当前有效状态，`changes/` 只在工作未完成期间保存当前事实、目标状态和主要变化。
- 新 story 一次性确定任务集合；已有 story 的任务文件身份固定，变化和代码错误回到原任务。
- `$cx-review` 连续执行交付物质量审查和完成证据门禁。
- 工作开始时先读取 `docs/cx/docs/` 专题文档并搜索已经登记的通用能力；研究结论保存到 `docs/cx/notes/`。
- 除非用户明确要求具体校验或异常处理，否则异常原样传播并中止程序；开发代码只保留当前入口和当前行为。
- 双语安装包、模板、说明、代理和流程校验器表达同一套当前规则。
- `$cx-tdd` 统一承担所有语言的测试先行主流程、断言纪律和真实数据原则；`$cx-pytorch-tdd` 与 `$cx-rust-tdd` 只保留语言特有补充。

## 主要变化

1. 正式文档从允许记录处理前后状态，改为只表达当前有效状态。
2. 变更文件从长期审计记录，改为当前工作指令；文件先提交到 Git，完成后删除并再次提交。
3. 已有 story 从可以继续拆出变更任务，改为保持原任务文件数量和身份不变，并原地修改任务正文。
4. 代码错误从直接修复或新增修复任务，改为先登记临时变更并回到对应原任务。
5. 交付审查从 `$cx-review` 和 `$cx-evidence` 两个技能，改为 `$cx-review` 内的两个连续阶段。
6. 通用能力发现从直接搜索源码，改为先读取 `docs/cx/docs/` 专题文档，再搜索已经登记的通用包和真实调用点。
7. 研究结论从只规定输出格式，改为保存到 `docs/cx/notes/`，并要求针对具体问题作通俗、流畅的说明。
8. 异常处理从允许预先校验、包装或业务兜底，改为默认禁止代理自行增加校验并抛出、掩盖、转换或兜底异常；底层异常必须保持原始类型、信息和调用栈并自然中止程序。只有用户在当前请求中明确要求的具体校验或异常处理才允许实现。
9. 开发实现从可能保留旧接口和兼容分支，改为只实现最新意图；旧入口、别名、适配层、桥接层、旧参数、旧配置、旧路径、旧行为和相关痕迹必须全部删除。
10. 测试技能从三个技能重复流程，改为 `$cx-tdd` 统一失败、通过、重构、审查和断言纪律；有序集合默认只检查首项、末项和一个中间项，期望侧只允许已确认常量。
11. Python 数据测试改为由 `tests/__init__.py` 一次读取测试数据库并共享真实对象；Rust 数据测试改为由统一夹具模块使用一次初始化机制共享真实对象。除非用户明确要求，两个语言技能都禁止模拟测试。
